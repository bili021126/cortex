## 总览：八处缺口，两处致命

凝光在 v2.5.12 的三项审计中发现了 **8 处缺口**。我逐一追踪它们在架构中的根系走向后，得出判断：

> **P0（2 处）**：系统架构的「根」出了问题——宪法演进能力和治理层有效性被锁死。不修复，整个治理层形同虚设。  
> **P1-P2（3 处）**：治理层的「枝干」有裂痕——影响运行健壮性，但不致命。  
> **P3-P4（3 处）**：治理层的「叶片」有缺口——精细化所需，当前阶段可暂缓。

## 🔴 发现 1-A：自反性缺口 —— 原则七的根系腐烂

### 缺口本质

原则七的六项子约束定义了修宪行为的完整规则，但**没有定义这些规则本身如何被修改**。凝光的描述精准：这是"镜子照镜子时镜子不能先把自己涂黑"——约束修宪的规则本身不可被修宪规则修改。

### 架构影响范围

| 影响层 | 影响方式 | 严重度 |
|--------|---------|--------|
| **宪法元规则层** | 原则七的六项子约束构成一个封闭的元规则集合。若未来需要新增子约束（如紧急修宪通道）、修正现有子约束语义、或调整优先级，当前宪法**无任何条款提供修改子约束本身的流程依据** | 🔴 致命 |
| **修宪自动化管线** | governance-loop.ts 中的 `judgeProposals()` → `evaluateAmendment()` → `applyApproved()` 流程已建成，但管线的元规则层是锁死的——六项检查项（principle-immutability / version-continuity / structural-consistency 等）本身不可演进 | 🔴 致命 |
| **DocGovernAgent** | 凝光的 system prompt 中写明了"你只提案不动宪法"，但元规则层锁死意味着凝光可以**发现缺口却无法修补补丁机制本身**——她发现了自反性缺口后生成的 AM-2025-0606-001，正是为了修这个元规则锁 | 高（但已被提案覆盖） |
| **昔涟评判引擎** | `amendment-judge.ts` 的 check 顺序硬编码为 `CHECK_ORDER` 常量数组。若原则七新增子约束7，评判引擎无法自动感知——需要手动更新评判逻辑 | 🟡 中 |
| **所有未来修宪提案** | 任何涉及修改原则七子约束的提案，在当前宪法下都**没有宪法依据可引用**，违反子约束1（"修改必须显式引用目标宪法条款"）的精神 | 🔴 致命 |

### 代码层面的根系追踪

```
原则七六项子约束（宪法文本）
  ├── governance-loop.ts: 修宪管线编排（loadPendingProposals → judgeProposals → applyApproved）
  │   ├── amendment-judge.ts: 六项检查项（硬编码 CHECK_ORDER，不可演进）
  │   ├── amendment-applier.ts: 文本替换执行器（纯字符串操作，无约束感知）
  │   └── 无 "子约束修改" 专项检查逻辑
  └── doc-govern-agent.ts: 凝光的 system prompt 声明 "你只提案不动宪法"
       └── 发现缺口 → 生成提案 → 但元规则层无演进入口
```

**关键发现**：`amendment-judge.ts` 的 `CHECK_ORDER` 是硬编码常量数组。如果未来新增子约束7，评判引擎不会自动新增对应的 check 函数。这是一个**元规则层的硬编码锁**——和宪法的自反性缺口是同一问题的代码面映射。

### 修复建议

AM-2025-0606-001 的解决方案是新增子约束7「子约束修改规则」，设定更严格的审查条件（双重把关、不可代理裁决、版本演进链追踪）。这是正确的方向。但代码层需要同步跟进：

1. `amendment-judge.ts` 的 `CHECK_ORDER` 需改为可扩展模式——新增子约束时应自动注册对应的 check
2. `loadPendingProposals()` 在筛选待决提案时，应增加对 `section` 字段含「原则七·子约束」的提案的**标记逻辑**，以便评判引擎识别并应用更严格的审查

## 🔴 发现 3-A：审计闭环缺失 —— 审计报告沉入雨林无人收

### 缺口本质

DocGovernAgent 的三大审计节点（plan_review / doc_audit / constitution_check）已落地，审计报告写入 `docs/auditing/`。但**审计报告产出后的处理流程未定义**——谁负责整改、如何验证整改、判例如何引用、未关闭的审计发现是否能进入下一阶段。

凝光自述："审计报告目前写完磁盘即结束，用户不知情"。

### 架构影响范围

| 影响层 | 影响方式 | 严重度 |
|--------|---------|--------|
| **治理层有效性** | 审计不闭环 = 审计不存在。DocGovernAgent 产出的审计报告如果无人响应，则治理层的核心职能——"确保系统与宪法一致"——无法落地 | 🔴 致命 |
| **DocGovernAgent 审计循环** | 凝光发现缺口 → 生成提案 → 写入审计报告 → **流程终止**。无"整改 → 验证 → 关闭"的后半段 | 🔴 致命 |
| **MemoryStore 治理分区** | doc-govern-agent.ts 的 `docGovernMemoryQuery()` 读取 Episodic + Knowledge 类型的 Active + Archived 态记忆，但审计结论的归档路径未正式入宪 | 🟡 中 |
| **阶段门禁** | 审计发现未关闭是否构成阶段门禁（Core-1→Core-2）的阻塞条件？当前宪法无此规定，可能导致带着未修复的宪法缺口进入下一阶段 | 🟡 中 |
| **判例引用机制** | 凝光的审计报告包含判例记录（如 NG-2026-0606-SelfRef-Gap），但宪法未定义判例的有效期和失效条件——判例何时作废？谁来决定？ | 🟡 中 |

### 代码层面的根系追踪

```
DocGovernAgent 审计流程（现况）
  读取 doc-govern-agent.ts system prompt
    → 发现缺口 → write_file 到 docs/auditing/<日期>-<标题>.md
    → 生成修宪提案到 docs/amendments/AM-*.json
    → 流程终止 ❌（缺下半段）

应有的审计闭环（缺失）
  DocGovernAgent 审计 → 写入 audit report
    → 通知整改责任人（谁？未定义）
    → 整改完成 → 验证（谁来验证？未定义）
    → 关闭审计发现 → 归档判例（如何归档？未定义）
    → 记入治理摘要（summarizeGovernance 已实现但只统计提案状态）
```

**关键发现**：`governance-loop.ts` 的 `summarizeGovernance()` 目前只统计**修宪提案**的状态（pending / approved / blocked / applied），完全不包含**审计发现**的关闭状态。这是一个代码层面的审计盲区——治理摘要不知道审计报告的闭环情况。

### 修复建议

1. 定义审计发现的生命周期状态机：`open → in_remediation → verified → closed`
2. 在 `governance-loop.ts` 中新增审计发现跟踪逻辑——与提案跟踪并列
3. 定义整改责任人的指派规则（默认是责任 Agent？MetaAgent 指派？开拓者裁决？）
4. 宪法新增条款：未关闭的 P0 审计发现不得进入下一阶段门禁

## 🟡 发现 2-A：DECISION_REQUIRED 回退机制缺失 —— 决策轨道没有安全阀

### 缺口本质

宪法 §8.2 定义 DECISION_REQUIRED 为"打断当前 UI → 弹出决策界面 → 阻塞等待用户响应"，但未定义用户不响应时发生什么。

### 架构影响范围

| 影响层 | 影响方式 | 严重度 |
|--------|---------|--------|
| **用户交互体验** | 用户忙碌/离线时，DECISION_REQUIRED 事件无超时处理——系统可能无限阻塞 | 🟡 中 |
| **ButlerAgent** | 管家的三路分发策略（FYI / WARNING / DECISION_REQUIRED）在 DECISION_REQUIRED 路径缺少超时和默认行为处理 | 🟡 中 |
| **ConfirmGate** | 确认门（L2/L3 不可逆操作）依赖 DECISION_REQUIRED 轨道。若确认门无超时处理，关键操作可能因用户无响应而挂起 | 🟡 中 |
| **Scheduler 任务编排** | 拓扑排序 → 逐层并行的调度策略，若某节点因 DECISION_REQUIRED 阻塞无超时，后续节点无法继续 | 🟡 中 |

### 修复建议

宪法需新增 DECISION_REQUIRED 的配套机制：
1. 超时阈值定义（建议：30分钟默认超时）
2. 超时后的默认行为（建议：自动驳回 + 降级为 WARNING 通知）
3. 重复 DECISION_REQUIRED 的防抖策略（同议题 24h 内不重复弹出）
4. 用户离线时的处理策略（排队等待 vs 降级为静默记录）

---

## 🟡 发现 1-B：紧急修宪通道缺失 —— 没有特快列车

### 缺口本质

六项子约束假设完整的提案→评判→裁决→写入→验证流程。紧急情况下（安全漏洞、数据丢失风险、关键路径阻塞），完整流程可能太慢。

### 架构影响范围

| 影响层 | 影响方式 | 严重度 |
|--------|---------|--------|
| **系统安全性** | 紧急安全漏洞需要快速修复宪法防御时，标准修宪流程（多轮审计+评判+裁决）可能延误 | 🟡 中 |
| **修宪自动化管线** | governance-loop.ts 的流程无"紧急模式"分支——所有提案走同一流程，无加速通道 | 🟡 中 |
| **治理层危机响应** | 治理层设计目前未定义危机模式下的治理规则 | 🟡 中 |

### 修复建议

宪法新增紧急修宪子约束（作为原则七的子约束8候选）：
1. 紧急触发条件（需明确列举，防止滥用）
2. 快速通道流程（削减哪些环节？至少保留哪些环节？）
3. 事后审查机制（紧急修宪 → 事后 48h 内必须经标准流程重新确认）
4. 滥用防范（虚假紧急声明的惩罚机制）

## 🟡 发现 2-B：治理事件接入路径未入宪 —— 凝光的判决走不出群玉阁

### 缺口本质

治理层设计确认 DocGovernAgent "只写磁盘不 emit 事件"，审计结论无法进入通知管线（三轨语义分层）。

### 架构影响范围

| 影响层 | 影响方式 | 严重度 |
|--------|---------|--------|
| **通知管线** | PipelineObserver 统一管道收集所有可观测事件，但审计结论不在其中——用户无法通过标准通知渠道感知审计发现 | 🟡 中 |
| **ButlerAgent** | 管家负责分发三轨通知，但审计结论不在输入源中——无法向用户主动报告审计发现 | 🟡 中 |
| **用户知情权** | 宪法原则一（确认在用户手里）的精神要求用户知晓重要审计发现。但当前审计结论写完磁盘即结束，用户不知情 | 🟡 中 |

### 修复建议

1. 宪法 §8.2 新增：DocGovernAgent 的审计结论应生成 DECISION_REQUIRED 事件（P0 发现）或 WARNING 事件（P1+ 发现）
2. PipelineObserver 管道新增审计事件类型（如 `audit_finding`）
3. ButlerAgent 三路分发策略新增审计结论的分发规则

## 🟢 发现 2-C：通知持久化追踪未入宪

是否已读通知、是否已决策、决策结果如何归档回 MemoryType.Governance，无宪法规定。

**影响范围**：低。不影响系统运行健壮性，影响的是审计追溯的完整性。  
**修复建议**：后续精细化阶段入宪即可。

---

## 🟡 发现 3-B：宪法与治理层设计层级冲突解决原则缺失

宪法将治理层设计定位为"配套文档"（§10），但未定义冲突时的裁决原则。

**影响范围**：中。目前未发生实际冲突，但随着治理层设计文档演进，冲突风险增大。  
**修复建议**：新增宪法条款——冲突时宪法优先，但须在宪法修正记录中标注冲突内容及裁决理由。

---

## 🟢 发现 3-C：阶段门禁宪法定义不完整

架构图中有"阶段门禁检查表"，但宪法未定义触发条件、通过/失败标准、补救流程。

**影响范围**：低。阶段门禁目前以软约束形式运行，Core-1 阶段暂未遇到门禁阻塞。  
**修复建议**：Core-2 启动前必须完成阶段门禁的宪法定义——这应设为 Core-2 的准入条件之一。

## 三句话结论

1. **两根腐烂的根须**：自反性缺口（发现 1-A）让宪法在元规则层锁死，审计闭环缺失（发现 3-A）让治理层有头无尾——这两处修复后才谈得上治理层有效运行。
2. **最紧迫的代码层操作**：昔涟评判 AM-2025-0606-001（已生成待决）+ 在 `governance-loop.ts` 中新增审计发现跟踪闭环逻辑。
3. **架构的健康信号**：雨林能自己发现自己的根系腐烂——凝光审计发现了这些缺口，本身就证明了审计机制的有效性。修完这两处 P0，Cortex 的治理层将进化到可自我维系和自我演进的状态。
