## 第一部分：概念设计与工程实践对照分析

### 一、对照总览

概念设计定义了 6 大交付类别和 5+1 退出标准。工程实际交付了 151 测试、10 包 monorepo、完整类型系统。存在三类偏离：可接受简化、设计缺陷、意外暴露。

### 二、交付物对照

#### 2.1 运行时
- ✅ 单进程 async 协程 + `setImmediate` 就绪扫描
- ⚠️ 图执行调度器：概念设计说"2-3 柱"，实际 5 包互联——接口传播半径超出预期
- ❌ CortexEngine 上帝类：10 职责（DI 容器、请求路由、事件订阅、计划执行、记忆写入、消息总线、blocked 恢复、状态/中止、诊断、确认流）
- ✅ CoroutineRunner（Think→Act→Observe）+ DeepSeekLLMInvoker

#### 2.2 记忆
- ✅ MemoryStore + 活跃/归档两态 + BFS 三跳 + 关键词旁路 + 取向过滤 + 私密隔离

#### 2.3 Committee
- ✅ CommitteeManager convene/dissolve/report 管线连通性验证
- ⚠️ 仅分歧交付路径（三级收束延后到 Core-1b）
- ❌ 固定成员制不足——需动态调整、责任链驱动

#### 2.4 确认门
- ✅ ToolGateway L0-L3 四级可逆性 + 染色校验 + run_shell 白名单 + 两阶段确认

#### 2.5 事件总线

- ⚠️ EventBus API 四方法签名初始不对称（已修复：统一为 `CortexEventType | (string & {})`）

#### 2.6 测试
- ✅ 151 测试全通过——超出概念设计基线 50 倍

## 第二部分：类型系统加固教训

### 一、事件回顾

三枚举全量迁移：`ToolStaining`（7 文件）、`MemoryType`（10 文件）、`CortexEventType`（5 文件）——总计 9 文件、30+ 处修复。

### 二、五条教训

**教训 1：`import type` 对 `as const` 枚举对象无效**
- `as const` 对象既是类型也是值。`import type` 只导入类型部分，运行时值为 `undefined`
- 修复：拆分为独立 value import
- 工程原则 P6：修改 shared 包类型后，需额外验证所有 `import type` 是否正确拆分

**教训 2：`CortexEventType` 是目录而非穷举**
- 将 `publish()` 签名改为 `type: CortexEventType` 拒绝了测试事件、自定义事件
- 修复：`CortexEventType | (string & {})`——编译期提示 + 运行时开放
- 工程原则 P5：EventBus 四方法使用相同的目录非穷举类型

**教训 3：接口升级的传播半径被严重低估**
- 两个枚举值的变更导致 17 处编译错误，跨越 9 个文件、4 个运行时包 + 测试
- 根因：shared 包中作为接口契约的类型，消费者不限于显式 import 的文件
- 工程原则 P1：修改前必须穷举全 monorepo 所有引用点

**教训 4：概念设计未将"类型系统"作为独立设计维度**
- TypeScript 的工程特性（`isolatedModules`、`as const`、联合类型坍缩）不是"纯实现细节"
- 建议：Core 阶段概念设计中应包含"类型系统的工程约束"作为独立讨论项

**教训 5：测试未覆盖类型迁移的语义一致性**
- 测试只验证了"代码能跑"，未验证"类型使用符合预期语义"
- 后续需验证：a) tsc 零错误 b) 全量测试通过 c) 字符串序列化/反序列化一致性

### 三、对后续阶段的指导

| 阶段 | 影响 |
|------|------|
| Core-1d（better-sqlite3） | schema 枚举值迁移需同步更新所有 INSERT/UPDATE 和反序列化 |
| Core-2a（19 柱上线） | P1 传播半径穷举从 9 文件扩展到 30+ 文件 |

## 第三部分：内外源复杂度分析

### 一、核心隐喻选择

**选定：Cortex = 有安全边界的 LLM 驱动工具链路。**

放弃大脑隐喻（无 Hebbian 学习、无突触可塑性）和国家/政府隐喻（三权不合一、Committee 无否决权）作为架构推导依据。

三条子原则：

| 原则 | 含义 |
|------|------|
| **安全网 > 智能** | 宁可拒绝不确定的操作，不可替用户做高风险决策 |
| **契约 > 涌现** | 组件之间用类型契约通信，不靠"希望 LLM 理解上下文" |
| **可组合 > 可自愈** | 管线可以拆解和重组，不追求系统自动修复自身错误 |

### 二、内外源复杂度量化

**外源**（问题域固有）：LLM 非确定性、不逆副作用、记忆检索精度 gap
**内源**（架构引入）：确认门伪异步、混合总线、静态模板规划、19 PillarId 分类通胀、Engine 上帝类、三层隐喻互斥

**内外源比例 1:2。** 架构自己制造的复杂度比问题本身的复杂度高一倍。

### 三、复杂度倒挂

```
高内源复杂度（本应简单）：
  ├── 确认门事件链路（本质：confirm("Are you sure?")）
  ├── 19 个 PillarId 类型定义（本质：3 种执行策略配置）
  ├── 8 个事件子类型（本质：通知管道 QOS 分级）

低复杂度区（本应深挖）：
  ├── 任务规划：4 节点固定模板，关键词匹配
  ├── 错误恢复：无重试，无降级，超时就死
  ├── 执行反馈：结果不回传规划器（断环）
  └── 记忆利用：无法验证 LLM 是否使用了记忆上下文
```

确认门是系统里最精雕细琢的子系统。而 LLM 根据工具结果动态调整策略——这才是工具链的真正价值——几乎不存在。

### 四、根本诊断

**三个具体症状**：过早收敛、安全焦虑驱动、分层不分解。

**如果从零重建**：确认门走同步调用、柱子是配置驱动策略、规划器先做 LLM 原生动态规划、引擎拆成 pipeline 中间件链、事件总线退化通知管道。

### 五、Core 阶段首要任务

**剥掉内源复杂度，让外源复杂度浮上来。**

优先级：
1. 确认门去事件总线化（改为同步调用）
2. 规划-执行闭环（执行结果回传，触发重规划）
3. 引擎职责拆分（pipeline 中间件链）
4. 柱子从类层次退化为配置驱动
5. 事件总线降级为通知管道

### 六、阶段判定

**Meso-Lite 不是失败的。** 27 个测试全绿，LLM→工具→确认门链路全线打通。

**但它是"过早架构化"的。** 在只有 3 根实际工作柱子、4 节点固定模板、单进程 CLI 的运行剖面下，建立了 19 柱分类体系、8 事件类型体系、三层隐喻文档体系。

Core 阶段不应是 Meso-Lite 的"扩建"——应是 Meso-Lite 的"蒸馏"。
