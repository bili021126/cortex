### 二、核心交付

#### 2.1 过渡-1a：拆 Engine 上帝类 + 立 3 根真柱

**为什么 3 根？** 3 根对应正交的三视角——Summarize（结构）→ Code（执行）→ Safety（安全）。3 根是最少可验证"多视角"的柱数量。

**三柱定义：**

| 柱 | 宪法来源 | 取向 | 核心行为 | 工具集 |
|----|---------|------|---------|--------|
| Summarize | 4.1.1 信息与知识皮层区 | 管家 | 读上下文摘要 → 输出结构级影响评估 | read_file（只读） |
| Code（交融模式） | 4.1.2 代码与逻辑皮层区 | 搭档主+管家辅 | 调研阶段（管家辅助）：检索→定位 / 编码阶段（搭档回归）：Think→Act→Observe | search_replace, create_file, run_in_terminal + 交融借用只读工具 |
| Safety | 4.1.3 质量与安全皮层区 | 监理 | 审查变更→风险判定→审计日志→确认门 | read_file, grep_code（只读） |

**从 Engine 切出：executeNode 的检索/生成/安全判定 → 各自 PillarRunner；processConfirm → ConfirmCoordinator**

**退出标准**：3 柱各独立完成 10 任务 + Engine 代码行数减 > 40%

#### 2.2 过渡-1b：Scheduler 职责回归

| 职责 | 当前所在 | 迁回后 |
|------|---------|--------|
| blockedNodes 恢复 | Engine | Scheduler.resumeBlocked() |
| nodeResults 记录 | Engine | Scheduler.nodeResults |
| abortSignal 传播 | Engine | Scheduler.abortCascade() |
| activeLocks 管理 | Engine | Scheduler.activeLocks |

**退出标准**：Scheduler 独立 500 任务无死锁，stuck 率 < 1%

#### 2.3 过渡-1c：最简 Committee 管线（3 柱）

触发条件：needsMultiPerspective / 风险 ≥ L2 / blocked / 跨模块影响

流程：Summarize 输出 → Code 输出 → Safety 输出 → DocGovern 记录 → Meta-Agent 裁决 → 决议写入节点

**监理优先原则**：Safety 柱权重大于 Code 和 Summarize。L3 默认驳回，除非有事实证据证明前提错误。

**退出标准**：10 个 Committee 场景全部通过，监理优先采纳率 ≥ 80%

#### 2.4 过渡-1d：ConfirmCoordinator 独立化

**退出标准**：50 次确认独立处理（20 L1 + 20 L2 + 10 L3），无遗漏、无重复

#### 2.5 过渡-1e：DocGovern 最简版

JSON 文件，结构：`{ committee_sessions: [{ session_id, task_id, trigger, pillars, discussion, verdict }] }`

**退出标准**：所有 Committee 会话的结构化记录完整，可被回归测试读取

## 第五部分：Core-2 架构演进方向（终审讨论记录）

> 终审确认过程中，明确了当前单树模式在复杂项目前的三大瓶颈及其演进方向。
> 以下属于 Core-2 设计储备，不在当前三阶段偿还范围内。

### 一、单树 → 项目森林（Project Forest）

当前 `TaskTree` 单树模式在前后端等复杂项目中会遭遇：
- 规划压力：MetaAgent 需一次性生成几十节点并理清跨域依赖
- 调度瓶颈：`topologicalSort` 将不相关节点（前端UI/后端数据库）错误串行化
- 记忆信噪比：多个域的经验记忆涌入同一 MemoryStore，检索精度下降

演进方向：
1. **物理隔离升级**：每个 Project 拥有独立 MemoryStore 命名空间，Agent 只在专业领域检索
2. **逻辑关联升级**：任务树之间通过跨项目事件通信（而非父子节点依赖），如 `project:backend:api-ready`
3. **领而不执升级**：MetaAgent 先生成高阶 Project Plan，再对每个子项目生成独立 TaskTree

### 二、结构化错误上报协议（“带卷宗上报”）

当前重规划链路存在信息衰减：Agent 失败报告 → Scheduler 封装 → MetaAgent 只看到“节点失败”而丢失“为什么失败”和“怎么修”。这是重规划泛化的根因。

演进方向：
- `NodeResult.error` 从 `string` 升级为结构化错误报告
- 包含字段：`errorCode` / `failingAgent` / `failingNodeId` / `suggestedFix` / `rawReport`
- MetaAgent 的 REPLAN_SYSTEM 基于完整卷宗做出精准修复决策，而非泛化分析

### 三、内存事件管道 → 持久化治理日志

当前 PipelineObserver 是内存环形缓冲区，进程退出即消失。复杂项目的跨会话事件链、Committee 追溯、审计查询均无法实现。

演进方向：
- 事件持久化到 MemoryStore（`MemoryType.Governance`）
- 支持事件关联查询（“node.failed 是否导致下游级联失败”）
- 跨会话事件链拼接

### 与三阶段策略的关系

上述三项属于 Core-2 架构演进，不纳入当前偿还范围。但以下地基打磨项为它们做了铺垫：
- Agent 继承统一 → 新 Agent 类型（如监理 Agent）可低成本插入
- Toolkit/ToolRegistry 合并 → 跨项目工具授权可走统一入口
- MAX_LOOPS 参数化 → 不同项目类型的 Agent 可定制执行策略

**原则**：当前阶段只打地基，不盖新楼。Project Forest 是 Core-2 的新楼层——地基不打平，新楼立不稳。
