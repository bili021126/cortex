## 第二部分：原型阶段验证——失败

### 结构性失败（6 项）

| # | 失败 | 证据 | 根因分类 |
|---|------|------|---------|
| **F1** | **19 柱定义，3 柱实现** | `PillarId` 类型有 19 个值，`pillar/src/pillars/` 只有 3 个文件。16 个柱子走 CoroutineRunner 通用回退 | 过早架构化——分类体系在实证数据不足时就固化为类型 |
| **F2** | **三取向模型被压缩为三份 maxIter 配置** | CodePillar(maxIter=15) vs SummarizePillar(maxIter=15) vs SafetyPillar(maxIter=15)：三者使用同一 ReAct 模板，差异仅在 systemPrompt。宪法所述的"强激活/中激活/弱激活/抑制"被压缩为 `getDefaultPillar()` 一行 switch-case | 宪法定义了神经架构级的激活模型，代码实现了参数级配置 |
| **F3** | **确认门——最复杂的子系统——本质是 `confirm("Are you sure?")`** | ConfirmCoordinator 54 行 + ToolGateway 68 行 + 4 轮测试修复 + Promise.race 双等待模式。而 LLM 动态重规划——工具链的核心价值——是 0 行 | 安全焦虑在不成比例的抽象层级上被工程化 |
| **F4** | **事件总线承载控制流，违宪** | CONFIRMATION_REQUIRED 事件携带 `confirmationId` 期待外部 resolve → Promise 两阶段。宪法 2.3："横向事件不带指令"——这条事件本身就是指令。宪法 4.5："工具调用走直接函数调用，不走脊髓"——但 ToolGateway 走的是事件总线的 Promise 链路 | 架构设计自我矛盾 |
| **F5** | **规划-执行断环** | `handleExecute()` 调用 `scheduler.execute()` 获得 ExecutionReport 后，report 只用于格式化 statusMsg 字符串。没有任何逻辑将执行结果回传给 MetaAgent 触发重规划 | 宪法 2.2"请求重规划"机制缺失 |
| **F6** | **10 包模块化是结构上的，不是语义上的** | 10 包 tsc -b 编译通过，但 771 行 CortexEngine 中：DI 容器 + 请求路由 + LLM 调用 + Committee 观点生成 + 工具注册 + 诊断聚合。`handleExecute()` 内部 switch-case 创建 PillarRunner 实例——Engine 知道 Pillar 的构造细节 | 包边界 = 文件边界，不是职责边界 |

## 第三部分：宪法的设计缺陷——修宪清单

以下缺陷不是"概念设计 vs 实现的偏离"——是宪法条文本身存在逻辑矛盾、工程可行性问题或覆盖盲区。其中已触发项立即修宪。

### 3.1 自相矛盾条款

#### A-1：原则三 vs 确认门事件（立即修宪）🔴

- **宪法原文**：2.3 "横向事件不带指令"，4.5 "工具调用走直接函数调用，不走脊髓"
- **代码事实**：`ConfirmCoordinator.createHandler()` 将 `CONFIRMATION_REQUIRED` 发布到 EventBus，附带 `confirmationId`，期待外部 resolve → Promise。这条事件的语义就是"请用户确认"——一条指令。
- **矛盾**：宪法说事件不带指令，代码中确认门事件是系统中最明确的指令。宪法说工具调用走直接函数调用，代码中确认门走了事件总线。
- **修宪方案**：二选一
  - **方案 A（推荐）**：修改宪法——确认门事件从"横向事件"中豁免，作为"竖向下行"（Meta-Agent → 用户）的特殊事件类型。同步修改代码：确认门走直接同步调用，事件总线只用于审计日志。
  - **方案 B**：保留事件总线路径，但宪法为"护栏类事件"新增例外条款，明确区分"指令性事件"（确认门、硬拦截）和"描述性事件"（状态变更、进度更新）。

#### A-2：原则四（不可逆操作需用户确认）vs 原则二（规划与执行分离）（讨论后修宪）🟡

- **宪法原文**：4.5 "Meta-Agent 不通过工具调用层执行任何操作"、"不可逆操作拦截逻辑在 Gateway 内，不在功能柱内"、"确认这个动作永远在用户手里"
- **代码事实**：Gateway 拦截不可逆操作 → ConfirmCoordinator → EventBus → Promise → 外部 CLI/桌面端 resolve。确认动作的"接收方"被分散在三个环节：Gateway（检测）、ConfirmCoordinator（管理生命周期）、外部（UI 层 resolve）。
- **矛盾**：宪法将"确认"定义为"用户动作"，但工程上确认是一个跨三个组件的时间窗口。当超时发生时，系统自作主张 resolve(false)——这本身是"替用户做决定"。
- **修宪方案**：宪法需定义"确认的完整生命周期契约"：
  - 确认窗口的语义（在此时刻内系统等待用户输入，超时不是"拒绝"而是"默认安全"）
  - 确认的传输通道（宪法承认这是特殊通道，不走普通事件流）
  - 确认超时的降级路径（明确的按风险分级的默认行为表）

### 3.2 工程不可行条款

#### A-3：19 功能柱分类体系在 Meso 阶段的可行性（立即修宪）🔴

- **宪法原文**：4.1.1——四大皮层区 18 柱 + LoopController，每柱有独立专业领域、每柱在三种取向下有不同激活状态。
- **代码事实**：3 个柱子使用同样的 ReAct 模板，差异仅在 systemPrompt。16 个柱子只有类型定义无实现。宪法描述的"取向激活矩阵"在代码中是 `getDefaultPillar()` 一行 switch-case + `isPillarSuppressed()` 一个布尔函数。
- **矛盾**：宪法以"19 个独立认知加工单元"为前提推导架构（Committee 组成、取向取舍、皮层区划分），但工程上只有 3 种执行策略。不是 19 种，是 3 种 × 参数化配置。
- **修宪方案**：将 4.1.1 从"19 个功能柱是固定设计"改为"4 个皮层区是抽象边界，每区的功能柱按阶段演化按需激活"：
  - Nano+：0 柱
  - Meso-Lite：3 柱（1 柱/区，选代表验证核心链路）
  - Core：按数据决定新增哪些柱
  - Full：完整激活
  - 同步删除 `PillarId` 中未实现的 16 个值，改为按需追加。

#### A-4：脊髓反射/第二意见反射在单进程下的物理意义（延后到 Core-2c）🟡

- **宪法原文**：2.3 分级响应——一级脊髓反射"不经过身份核心循环推理"，二级"低熵决策"，三级"完整推理"
- **代码事实**：InMemoryTransport 是一个事件数组。所有订阅者在同一个 Promise 链中回调。不存在"脊髓层"与"功能柱推理"的物理分离——它们在同一个调用栈中。
- **矛盾**：宪法描述的"脊髓自动响应不占用推理资源"在单进程 event loop 中没有对应物。一个回调的 CPU 时间就是占用了总时间片。
- **修宪方案**：在 2.3 中追加"在单进程模型中，分级响应的物理意义是执行优先级而非物理隔离。真正的物理隔离在 Core-2c 多进程/多线程阶段实现。"

### 3.3 覆盖盲区

#### A-5：规划执行闭环缺失（立即修宪）🔴

- **宪法原文**：2.2 "功能柱遇到节点内无法解决的阻塞必须请求重规划"——定义了重规划的"触发"，但未定义重规划的"流程"
- **代码事实**：`scheduler.execute()` 返回 ExecutionReport → `handleExecute()` 格式化 statusMsg → 结束。没有重规划路径。report.failedNodes 和 report.blockedNodes 被读入字符串但无任何后续处理。
- **修宪方案**：在 2.2 后追加"重规划闭环"条款：
  "当 ExecutionReport 包含 failedNodes 或 blockedNodes 时，Meta-Agent 重新生成 PlanningDirective——仅修改受影响节点的子任务，不重建已完成节点。重规划最多 3 轮，超限则交付用户裁决。"

#### A-6：Committee 权力边界空白（延后到 Core-2a）🟡

- **宪法原文**：4.1.3——Committee "只产出建议，不产出执行动作"
- **代码事实**：`runPostExecutionCommittee()` 生成报告附在 statusMsg 中返回给用户。Committee 既不能否决 Meta-Agent 规划，也不能阻止节点执行，甚至对"已执行的节点"发表事后观点——信息不对称。
- **矛盾**：Committee 的"建议"在时间线上晚于执行——等它开完会，代码已经写完了。
- **修宪方案**：4.1.3 需追加两条：
  1. "对于 needsMultiPerspective=true 的节点，执行必须在 Committee 讨论结束后启动，除非 Meta-Agent 行使紧急否决权"
  2. "Committee 对已执行节点的事后审查结果，Meta-Agent 必须在下一轮重规划中体现"

#### A-7：DocGovern 宪法地位空缺（立即修宪）🔴

- **宪法原文**：宪法第七章（记忆系统）定义了个体记忆，19 柱清单不含 DocGovern
- **代码事实**：`DocGovern` 已存在并工作（committee_sessions.json 42KB），Committee 已在调用它
- **修宪方案**：不是把 DocGovern 塞进 19 柱（它不是执行柱），而是在宪法中新增"组织记忆载体"的独立定义——一个跨取向、非执行的记录与检索角色，类似"史官"。宪法为它定义：存储范围、检索权限、与个体记忆的隔离边界。

### 3.4 宪法层级的设计哲学问题

#### A-8：宪法定义的"神经架构"与工程 "API 链"之间的抽象层级鸿沟（讨论后修宪）🔴

这是根本性的抽象层级问题，不是某一条款的问题。

- **宪法说**：Cortex 是"数字生命体"，Meta-Agent 是"前额叶"，功能柱是"皮层区柱"，Committee 是"挤到控制台前争论的情绪小人"
- **代码是**：`await this.metaAgent.classifyOrientation()` → `switch (pillarId) { case "CodeGen": new CodePillar(...) }` → `await llm.think()` → `await toolGateway.invoke()`
- **本质矛盾**：宪法以生物学隐喻推导架构（神经节点、脑干、脊髓、皮层区），所有结构分类源自"脑的哪个部分做什么"——但代码以工程约束推导架构（API 调用链、依赖注入、Promise 生命周期）。这两层之间没有映射规则。当宪法说"强激活"时，代码不知道强激活和弱激活在 ReAct 循环中的区别——都是 `maxIter=15`。

**修宪方案**：这不是修一条，是修宪法 v2.0 的方向性决策：
1. 如果宪法继续保留神经隐喻，则必须为每个隐喻概念定义工程映射——"激活" → CPU/API 调度策略，"抑制" → 工具集缩减规则，"Committee" → 时序约束
2. 如果宪法转向工具链隐喻（如架构反思纪要的结论），则神经术语降为启发式命名，架构推导从"脑的哪个部分"转向"管线的哪个阶段"

## 第五部分：阶段判定

**nano+ 到 Meso-Lite 的原型验证是成功的。** LLM→工具→确认门→Committee 四段链路全线打通，27 个真实 E2E 测试全绿。核心假设（LLM 可调工具、确认门可拦截不可逆操作、Scheduler 无死锁、ReAct 循环有效）全部通过实证验证。

**但宪法在原型验证中暴露了三个结构性缺陷：**

1. **过早分类**：19 柱体系在只有 3 个实现的情况下就固化为类型和文化
2. **隐喻越位**：神经架构的生物学隐喻推导出了代码无法兑现的抽象（脊髓反射的物理隔离、Committee "挤到控制台前"的同步协作）
3. **基础闭环缺失**：规划-执行-重规划的基础工程闭环没有被定义为宪法要求，却被定义为"神经节点的临时调度"——把工程必需当特性描述

**修宪的方向是削足适履的反面——不是把脚削小适应鞋，是承认鞋买大了，在脚长大之前先穿合适的尺寸。**
