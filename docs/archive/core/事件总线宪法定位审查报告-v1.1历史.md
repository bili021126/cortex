# 事件总线宪法定位审查——Meso-Lite 实证报告

> 审查对象：宪法第五章（通信架构）+ 宪法 2.3（原则三）+ 宪法 4.5（工具调用层）
> 审查方法：逐条宪法 vs 逐行源码 vs 全部 25 处 publish/subscribe 调用点
> 结论：宪法定义的四通道通信架构在当前代码中不存在。事件总线的实际形态与宪法定位之间存在结构性断裂。

---

## 一、宪法说了什么

宪法第五章定义了**四条通信通道**，各司其职：

| 通道 | 宪法职责 | 宪法约束 |
|------|---------|---------|
| **竖向总线** | Meta-Agent → Pillar 规划指令下行；Pillar → Meta-Agent 状态上行 | 仅在任务启动和异常仲裁时使用；不包含执行细节 |
| **横向总线（脊髓）** | 身份间状态同步、事件不带指令、"只描述发生了什么" | 发布者不知道订阅者；三级响应（反射/低熵/推理）；事件合并/关联/摘要 |
| **协作通道** | Committee 期间临时专用通信链路 | 只有参与 Committee 的功能柱可读写；格式为观点交换非指令下发；解散后立即销毁；记录归档到经验记忆 |
| **皮层区内部总线** | 功能柱间通信 | 功能柱通过内部总线接收协同邀请，确认参与后临时接入协作通道 |

宪法 5.4 末尾有一个明确的总线关系表，声明四条通道**不是同一个东西**。

## 二、代码实际是什么

### 2.1 只有一条物理传输

```typescript
// packages/event-bus/src/index.ts
export class InMemoryTransport implements Transport {
  private buffer: CortexEvent[] = [];  // 环形缓冲，容量 10000
  private subscribers: SubscriptionEntry[] = [];
  private wildcardSubscribers = [];
  
  async publish(event: CortexEvent): Promise<void> {
    this.buffer.push(evt);
    const matched = this.matchSubscribers(evt);
    await Promise.allSettled(matched.map(h => h(evt)));
  }
}
```

**整个 Cortex 只有一个 InMemoryTransport 实例**。宪法定义的四条通道全部映射到这一个对象。不存在独立的"竖向总线"对象、"脊髓"对象、"协作通道"对象、"皮层区内部总线"对象。

### 2.2 "竖向总线"不是事件总线

宪法定义的竖向总线（Meta-Agent ↔ Pillar 规划指令）**不走 InMemoryTransport**：

```
MetaAgent.createPlanningDirective()  →  返回 PlanningDirective (函数返回值)
CortexEngine.handleExecute()         →  调用 scheduler.execute(directive) (直接函数调用)
Scheduler.execute()                   →  返回 ExecutionReport (函数返回值)
```

这不是"竖向总线事件"，这是**同步函数调用的入参和返回值**。宪法 5.1 描述的"竖向总线"在代码中没有对应的 EventBus 实例——它被编译进了调用栈。

### 2.3 "协作通道"没有独立隔离

Committee 通信走的是**同一个** InMemoryTransport：

```typescript
// packages/committee/src/index.ts
private emit(type: string, payload: Record<string, unknown>): void {
  this.eventBus.publish({   // ← 同一个 eventBus
    type, publisher: "committee_manager", payload, ...
  });
}
```

宪法 5.4 要求的"只有参与 Committee 的功能柱有读写权限""Committee 外功能柱不感知"——在代码中不成立。`InMemoryTransport` 的 `subscribe()` 是基于 `type` 过滤的，不是基于"谁在 Committee 内"的。任何订阅了 `COMMITTEE_*` 事件类型的组件都能读到。

宪法要求的"协作通道在 Committee 解散后立即销毁"——在代码中不成立。事件进入环形缓冲区后保留到被覆盖，无法定向销毁。

### 2.4 "皮层区内部总线"不存在

代码中没有对应的物理通道。当前只有 3 个 Pillar，它们之间不通信。

### 2.5 "神经节点"不存在

CODE_AND_NEURAL_NODE_ACTIVATED 和 RESOLVED 事件类型定义了但没有任何代码发布或订阅它们。

## 三、实际流经事件总线的事件——按可靠性需求分

通过全量 `publish()` 和 `subscribe()` 调用点扫描（25 处 publish，25 处 subscribe），实际活跃的事件分为四类：

### 3.1 🔴 控制事件——携带指令，不容丢失

| 事件 | 发布者 | 语义 |
|------|--------|------|
| `control:confirmation_required` | ConfirmCoordinator | **"请用户确认此操作"**——携带 `confirmationId`，期待外部 resolve Promise |
| `control:confirmation_resolved` | ConfirmCoordinator | "用户已确认/拒绝" |
| `control:confirmation_timeout` | ConfirmCoordinator | "确认窗口超时" |

**宪法违规**：`CONFIRMATION_REQUIRED` 的语义是"请做某事"——这是指令，不是描述。宪法 2.3："横向事件不带指令"。宪法 4.5："工具调用走直接函数调用，不走脊髓"。

### 3.2 🟡 状态事件——需要但不紧急

| 事件 | 发布者 | 语义 |
|------|--------|------|
| `node:started/completed/failed/skipped/progress` | Scheduler | 节点生命周期状态变更 |
| `domain:pillar_*_completed` | 各 Pillar | Pillar 执行完成通知 |
| `domain:fusion_activated/dissolved` | FusionLifecycle | 交融生命周期 |
| `committee:convened/dissolved/divergence/report_ready` | CommitteeManager | Committee 生命周期 |
| `memory:written/read/state_changed` | MemoryStore | 记忆变更 |
| `system:startup/shutdown/heartbeat/abort` | 各组件 | 系统事件 |

### 3.3 ⚪ 领域事件——定义了但从未发布

| 事件 | 状态 |
|------|------|
| `domain:code_generated/reviewed/refactor_completed` | ❌ 无发布者 |
| `domain:security_issue_found/perf_issue_found` | ❌ 无发布者 |
| `domain:deploy_started/completed/failed` | ❌ 无发布者 |
| `sentinel:alert/heartbeat` | ❌ 无发布者 |
| `neural:activated/resolved` | ❌ 无发布者 |
| `system:replan_requested/planning_completed` | ❌ 无发布者 |
| `memory:cascade_frozen/obsoleted` | ❌ 无发布者 |
| `memory:private_access_detected` | ❌ 无发布者 |
| `control:irreversible_l3_blocked/pending` | ❌ 无发布者 |
| `committee:emergency_convened` | ❌ 无发布者 |
| `node:blocked/tool_executed` | ❌ 无发布者 |

**44 个事件类型中，24 个从未被发布**。它们只存在于 `CortexEventType` 的 `as const` 对象中——是文档，不是功能。

### 3.4 谁在订阅——几乎没有生产消费者

全量扫描发现：**生产代码中没有任何组件通过 subscribe() 消费事件来驱动控制流**。

唯一的订阅者：
- **测试代码**：通过 `bus.subscribe({ type: CET.NODE_STARTED })` 收集事件用于断言
- **ConfirmCoordinator 的间接订阅**：确认门不走 `subscribe()` 路径——它是 `publish()` + 外部通过 `resolve(confirmationId)` 直接调用（不走事件总线读取）

这意味着 **Meso-Lite 的事件总线是一个 write-only 系统**。事件被发布到环形缓冲区，但没有任何生产组件读取这些事件来做决策。这违背了 pub-sub 的基本定义——pub 存在，sub 不存在。

## 四、最大问题：确认门事件——宪法层面的自毁

这是整个事件总线设计中最严重的缺陷，直接影响 Core 阶段的架构决策。

### 确认门当前数据流

```
ToolGateway.invoke()                           // L2 工具到达
  → 检查 reversibility === "L2"
  → 调用 ConfirmCoordinator.createHandler()   // 创建回调
      → 生成 confirmationId
      → eventBus.publish(CONFIRMATION_REQUIRED) // 发布事件
      → return new Promise<boolean>(resolve => { // 挂起 Promise
            store resolver in Map<id, resolve>
            setTimeout(() => resolve(false), 120s) // 超时自动拒绝
          })
  → await 上述 Promise
  → 如果 resolved(true): 执行工具
  → 如果 resolved(false): 返回错误

外部 CLI/桌面端：
  → eventBus.subscribe({ type: CONFIRMATION_REQUIRED }) // 或轮询
  → 读取 confirmationId
  → 显示确认提示
  → 调用 engine.handleConfirmation(id, true/false)
      → ConfirmCoordinator.resolve(id, approved)
          → 从 Map 取出 resolve 函数
          → resolve(approved)  // ← 解开 ToolGateway 中的 Promise
```

### 问题清单

**1. 这不是 pub-sub，这是 RPC over EventBus**

`CONFIRMATION_REQUIRED` 携带 `confirmationId` 作为一个 rendezvous token。发布者（ToolGateway）等待特定的响应者（CLI/桌面端）通过 `confirmationId` 返回结果。这是请求-响应模式穿着发布-订阅的外衣。

宪法 2.3 的原则三被正面违背：
> "横向总线上流通的事件只描述'发生了什么'，不写'你该做什么'"

`CONFIRMATION_REQUIRED` 的语义是"你该做这个确认"。

宪法 4.5 的隔离声明被违背：
> "工具调用走直接函数调用，不走脊髓"

确认门走的就是脊髓。不是"通知有一个确认需求"，而是"通过脊髓传递确认请求并等待回应"。

**2. 同一个环形缓冲区承载完全不同可靠性级别的数据**

确认门事件和 `NODE_PROGRESS`（每轮 ReAct 循环发布一次）共用同一个 10000 容量的环形缓冲区。在大量节点并发执行的场景下，`NODE_PROGRESS` 事件可能填满缓冲区，导致 `CONFIRMATION_REQUIRED` 被覆盖丢失。

当前不会触发这个 bug 是因为：
- Meso-Lite 是单进程，确认门期间没有并发执行
- 10000 容量远超当前使用量

但这是**架构级的设计缺陷**——不同 QOS 的事件不应该共用同一个无优先级的环形缓冲区。

**3. 超时是"替用户做决定"**

```typescript
setTimeout(() => {
  resolve(false);  // 120 秒后自动拒绝
}, 120_000);
```

宪法 2.4：
> "无论信任等级多高，无论系统对人判断多准……确认这个动作永远在用户手里"

当超时发生时，系统单方面 `resolve(false)`——**这恰恰是替用户做了"拒绝"的决定**。宪法没有定义超时的语义（是默认安全还是默认通过），代码做了假设。

**4. 测试中需要 Promise.race + 轮询双等待**

```typescript
// 测试中等待确认事件的模式（4 轮修复后的最终形态）
const raceResult = await Promise.race([
  new Promise<CortexEvent | null>(resolve => {
    bus.subscribe({ type: CET.CONFIRMATION_REQUIRED }, (e) => { resolve(e); });
  }),
  new Promise<null>(resolve => setTimeout(() => resolve(null), 15000)),
]);
```

这不是"复杂功能需要复杂测试"——这是**设计制造的竞态**。测试在等待一个通过事件总线传递的同步请求——如果事件总线本身就是同步的（或者确认门走直接调用），就不需要 Promise.race。

## 五、根本诊断：宪法定义了一个不存在的系统

| 宪法概念 | 宪法定位 | 代码事实 | 差距 |
|---------|---------|---------|------|
| 竖向总线 | 独立的纵向信息通道 | 不存在——是函数调用的入参和返回值 | 宪法定义了架构概念，代码没有实例化 |
| 横向总线（脊髓） | 只描述事实、不带指令、分级响应 | 唯一的物理传输，承载了指令（确认门）和描述（进度） | 宪法约束被违背 |
| 协作通道 | 临时专用、隔离、Committee 解散后销毁 | 同一 InMemoryTransport，通过 type 过滤，无隔离，无销毁 | 宪法隔离约束无法兑现 |
| 皮层区内部总线 | 功能柱间通信 | 不存在 | 无对应代码 |
| 神经节点 | 冲突检测响应中心化协调 | 不存在 | 无对应代码 |
| 三级响应（反射/低熵/推理） | 按信息熵分级处理 | 不存在——所有订阅者以相同优先级回调 | 无对应代码 |
| 事件合并/关联/摘要 | 脊髓的整合功能 | 不存在 | 无对应代码 |
| 因果链验证/第二意见反射 | 关键事实事件交叉验证 | 不存在 | 无对应代码 |

**宪法第五章定义了 4 条通道和 8 项脊髓功能。代码只有 1 条通道、0 项脊髓功能、1 个违宪的控制事件。**

## 六、Core 阶段的决策：事件总线的三个方向

当前事件总线不能"修修补补"。问题不是某几个事件的语义需要调整——是整个宪法第五章描述了一个不存在的系统。Core 阶段必须做方向性选择：

### 方向 A：承认宪法超前，降级事件总线为通知管道（推荐）

**做法**：
1. 从 `Transport` 接口中删除 `CortexEventType` 作为类型参数——event bus 只传 `{ type: string, payload: unknown }`，不承担类型安全合约
2. 从 `CortexEvent` 中删除 `causalChain` 字段（没人用）
3. 确认门改为直接同步调用：`ToolGateway.invoke()` 调用 `ConfirmCoordinator.requestConfirmation()` 返回 Promise，不走事件总线
4. 事件总线只保留两类事件：
   - **进度通知**（NODE_PROGRESS、PILLAR_COMPLETED 等）——仅用于 UI 展示和日志
   - **审计事件**（MEMORY_WRITTEN、COMMITTEE_DISSOLVED 等）——仅用于事后追溯
5. 删除所有定义但未使用的 24 个事件类型
6. 修改宪法第五章：删除竖向总线/协作通道/皮层区内部总线/神经节点的独立通道定义；明确横向总线是"通知管道"，不是"控制平面"；删除脊髓整合功能（合并/关联/摘要/反射/第二意见）

**收益**：事件总线回归简单；确认门不依赖事件竞态；类型系统负担减半（44 → 20 事件类型）

### 方向 B：重做事件总线——从零实现宪法描述的四通道架构

**做法**：
1. 为四条通道分别创建传输实例——竖向总线（直接调用的薄封装）、脊髓（环形缓冲 pub-sub）、协作通道（隔离的临时通道）、皮层区内部总线（领域事件定向路由）
2. 确认门走竖向总线（同步请求-响应），不是脊髓
3. 实现脊髓整合功能——事件合并（同类型去重）、事件摘要（滑动窗口聚合）、反射规则引擎（模式匹配自动触发）
4. 神经节点实现冲突检测和临时中心化协调

**代价**：极高的实现复杂度。当前只有单进程 CLI，不需要协作通道隔离、不需要神经节点、不需要脊髓整合。建设这些功能在 Core 阶段缺乏运行时驱动力。

### 方向 C：保留现状，仅修宪法

**做法**：
1. 修改宪法第五章——删除代码中不存在的通道和功能
2. 将原则三改为"横向总线事件以描述为主，安全护栏类事件（确认门）例外"
3. 定义"当前阶段（Meso→Core）事件总线是一个统一的通知管道，通道隔离在后续阶段实现"
4. 代码不动

**代价**：宪法被削到适配代码而非指导代码——宪法变成了"事后合理化"而非"事前约束"。违背了宪法的根本目的。

---

## 七、推荐与下一步

**推荐方向 A**，理由：

1. **实证证据充分**：Meso-Lite 27 个 E2E 测试全绿证明了：不需要脊髓整合、不需要神经节点、不需要四通道隔离，核心链路就能工作。复杂的事件架构不是功能必须的。

2. **Core 阶段的第一优先级是"规划-执行闭环"**（修宪 A-5）。事件总线的过度工程化会分散 Core 阶段的注意力。

3. **确认门去事件总线化是最高优先级的可靠性改进**。Promise.race 双等待模式在 Core 阶段的并发场景下一定会出问题。

4. **方向 A 是可逆的**。如果 Core 阶段积累的运行时数据证明了脊髓整合或协作通道隔离的必要性，可以在 Core-2 阶段重新引入——届时将有数据而非隐喻驱动决策。

**不是放弃宪法。是让宪法从"定义幻想架构"变成"定义当前阶段的工程契约"。**
