# 治理层设计——Cortex 的政府

**版本**：v1.1（超前设计显式标注）
**定位**：宪法定义国家结构（大脑），本文档定义政府运行方式。二者分属国家/政府两个层级。
**关联**：Cortex 概念顶层设计 v2.4 第十章
**性质**：本文档不是蓝图——已落地部分写实现，超前设计部分标前提。政府可演进，本文档随治理层延伸程度逐版更新。
**来源**：合并自 `Core 阶段治理机制概念讨论.md`、`v2.0-治理架构深化讨论.md`、Meso-Lite 反思讨论

## 〇、国家/政府分层

宪法 v1.1 犯过一个错误：把政治隐喻（三省六部）和神经隐喻（大脑）塞进同一份文档。v2.0 剥离二者，但未给出治理层独立设计。

本阶段的认知突破：**国家 = 大脑，政府 = 治理层**。

| 层 | 隐喻 | 对应 | 性质 |
|----|------|------|------|
| **国家** | 大脑结构 | 宪法——Agent 类型、PipelineObserver、MemoryStore、Scheduler | 结构相对稳定，修需修宪 |
| **政府** | 运行方式 | 本文档 | 可演进、可试错、可依治理层延伸程度迭代 |

宪法只管国家结构，治理层设计管政府运行。政府可以推翻重来——国家结构不必跟着动。

## 第一部分：已落地

以下治理机制已有代码，在当前工程现实中运作。

### 1.1 SafeErrorReporter——错误上报的三档标准

建于 PipelineObserver 之上，是治理层的第一个工程锚点：

| 严重性 | 行为 | 已落地 |
|--------|------|--------|
| fatal | CRITICAL 同步 emit，阻止继续执行 | ✅ |
| degraded | HIGH 异步 emit，降级继续 | ✅ |
| silent | NORMAL 计数累加，≥3 次自动升级为 degraded | ✅ |

注入方式：`BaseAgent.setSafeReporter()` + `LlmAdapter.setSafeReporter()`，bootstrap 上层统一注入。**杜绝静默吞错**是治理层在当前工程阶段的核心约束力。

### 1.2 PipelineObserver——emit-only 单向管道

当前实现：`emit()` → `handler(event)`，handler 返回 void，emit 丢弃返回值。不存在 `emitAndWait()` 或 request-response 模式。**单向广播，无法形成控制回路。**

这个设计在架构上兑现了"巡视组"的雏形——看得见问题，拦不住执行。但 SafeErrorReporter 的 fatal 级走同步 emit 跨越了"看"与"拦"的边界——这是当前治理层唯一的干预力。

### 1.3 DocGovernAgent——三大审计节点

在 TaskBoard 上认领三种治理节点：

| 节点 | 职责 | 已落地 |
|------|------|--------|
| `plan_review` | 审查 MetaAgent 规划的自洽性 | ✅ |
| `doc_audit` | 阶段门禁时全量文档对照检查表打钩 | ✅ |
| `constitution_check` | 宪法一致性验证 | ✅ |

产出审计报告 → 写入 DocGovern 分区。

### 1.4 事轴/权轴的方向性——在执行闭环中已有雏形

**事轴（命令流，自上而下）**：用户意图 → MetaAgent 规划 → TaskBoard → Agent 执行。这是系统的主循环，代码完整。

**权轴（约束流，自下而上）**：当前以 SafeErrorReporter 的 fatal emit + Scheduler 的重规划（Level 2/3）为雏形。Agent 失败 → MetaAgent.requestReplan → 重规划最多 3 轮 → 超限交用户。约束往上报的通道已经存在，但仅限于失败场景——主动发现风险的"上报义务"尚不存在。

**横切监督方向**：PipelineObserver 独立于执行循环，DocGovernAgent 独立于 MetaAgent——监督不参与命令流的方向已经确立，但监督链的中间环节（常设委员会、TrustModel）空缺。

## 第二部分：设计锚点

以下方向性概念在当前工程中已有部分体现，作为后续治理层延伸的锚点。**需要治理层延伸程度进一步提升后才能展开为完整设计。**

### 2.1 委员会方向——治理审计 vs 执行裁决

**当前锚点**：needsMultiPerspective 节点是多视角并行审查的工程锚点——多个 Agent 并行认领同一节点，各产各的，用户收束。

**方向**：委员会按领域切分——

| | 常设委员会（方向） | 临时委员会（方向） |
|---|---|---|
| **领域** | 治理审计 | 执行裁决 |
| **主导** | DocGovernAgent | MetaAgent |
| **MetaAgent 参与** | 不入 | 主持召集 |

**当前延伸程度**：DocGovernAgent 已经在做治理审计（plan_review/doc_audit），但"常设委员会"作为一个多 Agent 审议实体不存在。临时委员会方向与 needsMultiPerspective 节点存在对应关系，但"裁决"语义尚未建立。

**延伸前提**：Committee session 机制（多 Agent 并行审议 + 决议记录）在工程中落地。

### 2.2 监理方向——从取向开关到合规审查

**当前锚点**：宪法 4.1.1 定义监理为取向开关——质量与安全皮层区在监理取向下激活。

**方向**：监理在政府层承担独立合规审查——合宪放行，违宪驳回。

**当前延伸程度**：监理没有独立实体。没有独立 LLM 通道。没有封驳权。取向开关 ≠ 独立 Agent。

**延伸前提**：监理从取向开关升级为独立 Agent 实体，拥有独立 LLM 调用通道。

### 2.3 上报方向——从失败处理到主动上报

**当前锚点**：重规划机制（Level 2/3）已落地。MetaAgent.requestReplan 在 Agent 失败后触发。

**方向**：四层上报——Agent 自愈 → 同级协商 → 重新调度 → 重规划 → 用户裁决。

**当前延伸程度**：Level 0（Agent 自愈）不存在。Level 1（同级协商）依赖不存在的临时委员会。Level 4（用户裁决）存在但仅通过重规划超限触发——不是主动上报。

**延伸前提**：Agent 自愈逻辑 + Committee session 机制。

### 2.4 通知管线方向——从单管混流到按类型分层路由

**当前锚点**：PipelineObserver 是唯一事件通道，ButlerAgent 订阅全部三级优先级，在 `_formatCritical` / `_formatLifecycle` 中用 switch-case 区分事件类型。

**当前问题**：所有通知语义搅在同一根管子里——生命周期事件（FYI）、执行失败事件（警告）、治理审计结果（需决策）全部走同一 `_onCritical` / `_onHigh` 路径。当前事件类型仅有七八种，ButlerAgent 的 switch-case 约十几行——耦合代价尚低。但每增加一种治理事件类型，都在给这根管子增肥。等到事件类型涨到三十种再来拆，每个 emit 点要改、ButlerAgent 的分发要拆、ConfirmGate 要接——改造成本指数增长。

**方向**：在 PipelineObserver 和 ButlerAgent 之间加一层语义路由——不改行为，只做类型分层。

| 通知类型 | 性质 | 用户动作 | 当前处理 | 当前事件示例 |
|---------|------|---------|---------|------------|
| `FYI` | 信息告知 | 看一眼/忽略 | console.log ✅ | node.start, node.complete, scheduler.layer.start |
| `WARNING` | 异常警告 | 可能需要介入 | console.log ✅ | node.failed, node.replan, error.* |
| `DECISION_REQUIRED` | 治理呈报 | **必须响应** | ❌ 不存在 | governance.audit.complete（预留） |

**当前延伸程度**：
- 通知类型分层**不存在**——所有事件走同一分发路径
- 治理事件类型**不存在**——`governance.audit` 系列事件在代码库中为零
- ConfirmGate 和 ButlerAgent 是两条独立路径——ConfirmGate 走 `bridge.confirm()`，ButlerAgent 走 `bridge.notify()`。治理事件如果走 ButlerAgent → notify，就跳过了 ConfirmGate。如果不走 ButlerAgent，就没有持久化追踪（用户是否已读？是否已决策？）
- DocGovernAgent 的三管线设计（执行/治理/通知）中，通知管线仅存在于设计文档——审计报告目前写完磁盘即结束，用户不知情

**架构预留方案（当前可执行，约 30 行改动，不改行为）**：
1. `ObservableEvent` 新增 `notificationType?: 'FYI' | 'WARNING' | 'DECISION_REQUIRED'` 字段——不强制订阅者处理
2. ButlerAgent 内部按 notificationType 分发——`_onFyi` / `_onWarning` / `_onDecision`，当前所有事件先标 `FYI`
3. 治理事件接入时不需要拆管道——直接往预留槽位 `DECISION_REQUIRED` 插，走 ConfirmGate 路径

**为什么越晚越贵**：现在事件类型一只手数得过来。后面每加一个治理事件，不拆就继续堆进 switch-case。等类型涨到三十个再拆——每个 emit 点、每个订阅者、每条测试链都要改。不是线性增长。

**延伸前提**：
1. DocGovernAgent 能 emit 结构化治理事件（当前只写磁盘，不 emit）
2. ButlerAgent `DECISION_REQUIRED` 路径接入 ConfirmGate（用户决策闭环）
3. 治理通知持久化追踪（已读/已决策/决策结果归档回 MemoryType.Governance）

## 第三部分：超前设计

> ⚠️ **以下内容属于概念设计论证阶段，不是当前工程计划。每项标注实现前提——治理层延伸程度必须到达对应门槛后，才能从"概念"进入"设计"，从"设计"进入"实现"。**

### 3.1 委员会体系完整定义

> **状态**：概念设计论证阶段
> **源文档**：`v2.0-治理架构深化讨论.md` 第一、二节

**概念定义**：

| | 常设委员会 | 临时委员会 |
|---|---|---|
| **领域** | 治理审计（Governance Audit） | 执行裁决（Execution Arbitration） |
| **触发** | 持续巡逻，主动发现 | 事件驱动，按需召集 |
| **输出** | 审计报告、修宪建议、技术债清单 | go/no-go 决定、升级/降级判断 |
| **主导** | DocGovernAgent | MetaAgent |
| **用户参与** | 大事必须在场，小事可不参与 | 按上报层级决定 |

**成员**：
- 常设委员会：DocGovernAgent（主导）+ 用户（常任决策）+ ReviewAgent/AnalysisAgent（按需）。MetaAgent **不入**。
- 临时委员会：MetaAgent（召集）+ DocGovernAgent（列席）+ 相关 Agent + 用户（按层级）。

**实现前提**：
1. 常设委员会和临时委员会作为 Committee session 机制在 engine 中存在
2. MetaAgent 有能力召集、Agent 有能力参与审议
3. DocGovern 分区能存储结构化 Committee 决议

### 3.2 纪检委监督链

> **状态**：概念设计论证阶段
> **源文档**：`v2.0-治理架构深化讨论.md` 第四、七节

**概念定义**：PipelineObserver（巡视组）→ DocGovernAgent（立案审查）→ 常设委员会（审理室）→ TrustModel/ConfirmGate（处分落地）→ 用户（最终决定权）。

核心原则——查办分离：PipelineObserver 只能 emit 不能 intervene；DocGovernAgent 能审查不能惩罚；ConfirmGate 是物理关卡不是审查者；用户是最终决定权。

**实现前提**：
1. 常设委员会存在（见 3.1 前提）
2. TrustModel 基础设施存在
3. ConfirmGate 能接收治理层的处分指令（当前只能做 L0-L3 拦截）

### 3.3 监理独立实体与封驳权

> **状态**：概念设计论证阶段
> **源文档**：`Core 阶段治理机制概念讨论.md` 第二、四节

**概念定义**：监理 = 门下省。合宪放行，违宪封驳。监理不做执行、不参与规划、不起草决议。监理连拦三次 → 常设委员会表决撤换 MetaAgent。

监理需要独立 LLM 通道（key-D），不占 Committee 配额。

**实现前提**：
1. 监理从取向开关升级为独立 Agent 实体（需要新的 AgentType 或独立运行时角色）
2. 监理拥有独立 LLM 调用能力
3. 常设委员会存在（撤换 MetaAgent 的表决机构）

### 3.4 TrustModel / TrustAgent

> **状态**：概念设计论证阶段
> **源文档**：`v2.0-治理架构深化讨论.md` 第九节

**概念定义**：TrustModel（确定性记分簿）负责自动衰减和事件驱动分数变更。TrustAgent（LLM 评估者）负责定性评估 Agent 行为 + 推荐最优 Agent（铨选）。

**实现前提**：
1. TrustModel 基础设施在 Core-2 中上线
2. Agent 行为数据积累到足以支撑定性评估

### 3.5 吏部/户部/礼部

> **状态**：概念设计论证阶段
> **源文档**：`Core 阶段治理机制概念讨论.md` 第二、五节

**概念定义**：吏部管柱的生命周期（激活/冻结/降权），户部管资源配额（token/context window），礼部管阶段关卡和文档规范。

**当前前提完全不成立**：所有 Agent 是代码中写死的类，没有一个走提名/激活/冻结流程。多柱体系不存在。没有资源预算机制。吏部/户部/礼部在治理层有独立 Agent 实例之前只能是概念。

**实现前提**：多柱体系上线 + 常设委员会存在 + 资源分配存在可调节参数。
