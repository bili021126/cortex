# 功能柱降级修正方案——从"19 柱类层次"到"策略配置 + 拓扑差异"

> 基于实证：CodePillar 有价值（双阶段拓扑），SummarizePillar/SafetyPillar 是 CoroutineRunner + 参数封装，16 个 PillarId 是字符串常量。
> 原则：拓扑不同→独立类，参数不同→配置，名字不同→类型占位。

---

## 一、修正目标

| 修正前 | 修正后 |
|--------|--------|
| 19 个 `PillarId` 字面量类型 | 保留 19 个 PillarId（作为领域概念，用于 Committee 身份、事件 publisher），但只要求 4 个有对应类 |
| 3 个 Pillar 类 + CoroutineRunner 兜底 | 1 个架构级 Pillar（CodePillar）+ 1 个配置驱动的 GenericPillar + 20 个 PillarId 占位符 |
| `PillarId` 类型定义在 shared | 不变——作为领域词汇保留，描述"Who"而非"How" |
| CortexEngine 中 switch-case 逐个创建 Pillar | Engine 调用 `PillarFactory.create(pillarId, config)`，内部查表 |
| SummarizePillar 类 (123 行) | → `PillarConfig` 对象（15 行） |
| SafetyPillar 类 (177 行) | → `PillarConfig` 对象 + `SafetyPostProcessor`（30 行——仅在需要 extractRiskLevel 时） |
| CoroutineRunner 作为"兜底" | CoroutineRunner 是唯一的执行器——所有配置路由到它 |

---

## 二、修正后架构

```
                 PillarFactory.create(pillarId, config)
                          │
                          │ 查配置表
                          ▼
              ┌───────────────────────────┐
              │ pillarId === "CodeGen"     │─→ CodePillar (双阶段拓扑)
              │ pillarId === "SecReview"   │─→ GenericPillar + safetyConfig + SafetyPostProcessor
              │ pillarId === "Summarize"   │─→ GenericPillar + summarizeConfig
              │ 其余 16 个 pillarId        │─→ GenericPillar + defaultConfig(pillarId)
              └───────────────────────────┘
                          │
                          ▼
                   CoroutineRunner
                   (唯一的 Think→Act→Observe 引擎)
```

### 修改点清单

#### 修改 1：新增 `GenericPillar`（替代 SummarizePillar + SafetyPillar + 16 个兜底）

```typescript
// packages/pillar/src/pillars/GenericPillar.ts（新文件，~50 行）

export class GenericPillar implements PillarRunner {
  public readonly pillarId: PillarId;

  constructor(
    pillarId: PillarId,
    private config: PillarConfig,
    private toolInvoker: ToolInvoker,
    private llmInvoker: LLMInvoker,
    private eventBus?: Transport,
  ) {
    this.pillarId = pillarId;
  }

  async execute(node: TaskNode, context: ExecutionContext): Promise<NodeResult> {
    // 1. 如果配置了工具白名单，包装 ToolInvoker
    const filteredInvoker = this.config.allowedTools
      ? new WhitelistToolInvoker(this.toolInvoker, new Set(this.config.allowedTools))
      : this.toolInvoker;

    // 2. 创建 CoroutineRunner
    const runner = new CoroutineRunner(
      this.pillarId,
      filteredInvoker,
      this.llmInvoker,
      this.config.maxIterations,
    );

    // 3. 注入系统提示到节点描述
    const enhancedNode: TaskNode = {
      ...node,
      description: `${this.config.systemPrompt}\n\n${node.description}`,
    };

    // 4. 覆盖取向
    const enhancedCtx: ExecutionContext = {
      ...context,
      orientation: this.config.orientation,
    };

    // 5. 执行
    const result = await runner.execute(enhancedNode, enhancedCtx);

    // 6. 后处理
    let extraPayload: Record<string, unknown> = {};
    if (this.config.postProcess) {
      const processed = this.config.postProcess(result.output);
      if (processed.enrichedOutput) {
        result.output = processed.enrichedOutput;
      }
      if (processed.extraPayload) {
        extraPayload = processed.extraPayload;
      }
    }

    // 7. 发布完成事件
    if (this.eventBus) {
      this.eventBus.publish(
        EventBuilder.ofType(this.config.completionEvent)
          .from(this.pillarId)
          .withPayload({
            nodeId: node.id,
            success: result.success,
            iterations: result.iterations,
            ...extraPayload,
          })
          .build(),
      );
    }

    return result;
  }

  abort(): void { /* GenericPillar 不持有 runner 引用，abort 由 Scheduler 通过外部标记处理 */ }
  getReActState() { return null; }
}
```

**注意**：`abort()` 的空实现是个已知问题。SummarizePillar 和 SafetyPillar 当前也是空实现。修正时在 `GenericPillar` 内部持有一个 `aborted` 标志 + runner 引用，或让 `abort()` 通过 `AbortController` 传播到 LLM 调用。**这是 Core 阶段解决的问题，不在本次方案范围。**

#### 修改 2：新增 `WhitelistToolInvoker`

SummarizePillar 的 `ReadOnlyToolInvoker`、SafetyPillar 的 `SafetyToolInvoker`、CodePillar 的 `FilteredToolInvoker`——三个类做同一件事：按白名单过滤工具。

统一为一个：

```typescript
// packages/pillar/src/index.ts

export class WhitelistToolInvoker implements ToolInvoker {
  constructor(
    private inner: ToolInvoker,
    private allowedNames: Set<string>,
  ) {}

  async invoke(tool: Tool, params: Record<string, unknown>): Promise<ToolResult> {
    if (!this.allowedNames.has(tool.name)) {
      return { success: false, error: `工具 "${tool.name}" 不在白名单中` };
    }
    return this.inner.invoke(tool, params);
  }

  getTool(name: string): Tool | undefined {
    return this.allowedNames.has(name) ? this.inner.getTool(name) : undefined;
  }

  listTools(): Tool[] {
    return this.inner.listTools().filter((t) => this.allowedNames.has(t.name));
  }
}
```

#### 修改 4：新增 `PillarFactory`

```typescript
// packages/pillar/src/pillar-factory.ts（新文件，~40 行）

export class PillarFactory {
  /**
   * 创建 PillarRunner。
   * 只有 CodePillar 需要特殊的双阶段拓扑——其他的都是 GenericPillar + 配置。
   */
  static create(
    pillarId: PillarId,
    toolInvoker: ToolInvoker,
    llmInvoker: LLMInvoker,
    eventBus?: Transport,
    fusion?: FusionLifecycle,
  ): PillarRunner {
    // CodePillar 有独特的双阶段拓扑——不能降级为配置
    if (pillarId === "CodeGen" || pillarId === "Refactor") {
      return new CodePillar(toolInvoker, llmInvoker, eventBus, fusion);
    }

    // 其余 17 个柱子走 GenericPillar + 配置
    const config = PILLAR_CONFIGS[pillarId];
    return new GenericPillar(pillarId, config, toolInvoker, llmInvoker, eventBus);
  }
}
```

**注意**：`Refactor` 也走 CodePillar——因为它和 CodeGen 共享相同的双阶段拓扑（调研→修改）。如果未来 Refactor 有独特拓扑，再独立成类。

#### 修改 5：SafetyPillar 的后处理独立为纯函数

```typescript
// packages/pillar/src/post-processors.ts（新文件，~25 行）
// 或直接放在 pillar-registry.ts 中

function safetyPostProcess(output: string): {
  extraPayload?: Record<string, unknown>;
  enrichedOutput?: string;
} {
  const riskLevel = extractRiskLevel(output);
  return {
    extraPayload: {
      riskLevel,
      auditEntry: {
        timestamp: Date.now(),
        riskLevel,
        reasoning: output.slice(0, 500),
      },
    },
  };
}

function extractRiskLevel(output: string): "L1" | "L2" | "L3" {
  const match = output.match(/RISK_LEVEL:\s*(L[123])/i);
  if (match) {
    const level = match[1].toUpperCase();
    if (level === "L1" || level === "L2" || level === "L3") return level;
  }
  if (/风险.*[高高严重不应做驳回]/.test(output)) return "L3";
  if (/风险.*中|缓释|谨慎/.test(output)) return "L2";
  return "L1";
}
```

## 六、不降级的内容

**CodePillar 不解散。** 它的双阶段架构（调研 → 编码）是真实的执行拓扑变化。两个 CoroutineRunner 串行、FusionLifecycle 状态管理、调研结果注入编码上下文——这不是 `PillarConfig` 能表达的。CodePillar 保留为唯一的架构级 Pillar 类。Refactor 柱复用 CodePillar 的拓扑。

**PillarId 类型保留 19 个值。** 作为领域词汇，它们承载了三个语义：
1. Committee 成员身份（"以 SecReview 的视角发言"）
2. 事件发布者标识（NODE_PROGRESS 的 `publisher` 字段）
3. 未来扩展点（Core-2a 阶段按数据决定激活哪些柱）

保留 PillarId 类型不变，但宪法和文档不再声称 19 个柱子是"19 个独立的认知加工单元"。
