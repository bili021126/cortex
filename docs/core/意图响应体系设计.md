# 意图响应体系——未来演进设计

**版本**：v1.1（务实收敛版）
**定位**：从极度中心化的 MetaAgent 单步拆解，演进为多轮澄清+标签匹配增强+自适应派发的渐进式协作模式
**关联**：Cortex 概念顶层设计 v2.5.4 第四章（甘雨战术中枢）
**性质**：纯概念设计——不动代码、不修宪。为后续 Core-2/Core-3 演进预留设计空间。
**对标**：治理层设计.md——超前设计部分，标注实现前提
**v1.0→v1.1 变更**：废弃"响应市场"隐喻——澄清是对话能力，匹配是排序能力，两者都不需要市场。市场是外来的重隐喻，澄清和匹配是内生的轻提升。

---

## 一、当前架构的集中化瓶颈

当前链路是一条单向管道：

```
用户 → ButlerAgent（中转）→ MetaAgent 甘雨（一锤拆解）→ TaskBoard → Scheduler 中心调度 → Agent 执行
```

### 三个集中化瓶颈

| # | 瓶颈 | 表现 | 后果 |
|---|------|------|------|
| 1 | **甘雨一锤定音** | 没有多轮澄清，拆解即发布。用户意图模糊时无结构化对话机制。 | 拆解质量依赖单次 prompt 运气。模糊意图 → 模糊任务树 → 低质量产出。 |
| 2 | **Scheduler 中心化** | 拓扑排序+逐层派发，Agent 被动等任务，无自主认领。 | 单一故障点。Agent 无主动性——来了就干，干完就走。无法按专长和信任择优。 |
| 3 | **TaskBoard 是命令板** | MetaAgent 写、Scheduler 派、Agent 执行——纯命令流，无价值信号。 | 没有激励信号。不知道谁做的好、谁空闲、谁该被优先分配。任务与 Agent 之间没有双向选择。 |

> 当前架构在小规模单用户场景下自洽。但随着 Agent 种类增长、任务复杂度上升、跨会话连续性需求出现，集中化瓶颈会从隐性转为显性。

## 二、两种组织模式：规划驱动 vs Vibe Coding

在进入具体设计之前，必须区分两种根本不同的组织基座：

| 维度 | 规划驱动（当前 Cortex） | Vibe Coding（未来可能） |
|------|---------------------|---------------------|
| **组织形态** | 树状——甘雨拆解→TaskBoard→按依赖执行 | 网状——Agent 共享上下文自由探索 |
| **信息流** | 自上而下——命令流 | 扩散式——Agent 互相看见彼此在做什么 |
| **调度方式** | Scheduler 拓扑排序 | Agent 自组织、自由加入 |
| **适合场景** | 明确需求、可拆解任务 | 探索性开发、原型迭代 |
| **当前状态** | Core-1 已落地 | 需 Agent-to-Agent 通信通道，不在当前路线图 |

**关键认知**：Cortex 的整个架构——MetaAgent 拆解、TaskBoard 认领、Scheduler 派发——全部建立在规划驱动模式之上。本文档的三层渐进设计（澄清→匹配增强→自组织）同样服务于规划驱动模式。Vibe Coding 是另一种完全不同的组织基座，需要 Agent 间双向消息通道、共享上下文总线等基础设施——这些不在本文档范围内。

## 三、务实三层渐进模型

```
┌─────────────────────────────────────────────┐
│  第一层：意图澄清层（Intent Clarification）      │
│  甘雨 + 用户多轮对话 → 已确认意图包              │
│  [scope清晰] [验收标准明确] [标签完整] [优先级]    │
│  本质：对话能力——纯 prompt 工程                  │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│  第二层：标签匹配增强（Tag Matching Enhancement） │
│  已确认意图包 → Scheduler 按信任权重+负载排序     │
│  在现有 _findMatchingAgent 中注入 TrustModel    │
│  本质：排序能力——Scheduler 内部增强              │
└──────────────────┬──────────────────────────┘
                   ▼
┌─────────────────────────────────────────────┐
│  第三层：自组织派发（Self-Organizing Dispatch）   │
│  Agent 监听能力匹配事件 → 自主 claim               │
│  → 圆桌协商时序 → 编队并行执行                    │
│  前提：Agent-to-Agent 通信通道 + TrustModel 上线   │
│  本质：自治能力——Core-3 远期目标                  │
└─────────────────────────────────────────────┘
```

**核心理念转换**：从"中央计划"走向"知情调度"。甘雨不再只是拆解——她先澄清意图，让意图更准；Scheduler 不再只是标签匹配——它按信任和负载排序，让分配更对；未来 Agent 不再只是被动等任务——它们基于事件自己 claim。

## 四、第一层：意图澄清层（Core-2a，纯 prompt 工程）

### 4.1 当前 vs 未来

| 维度 | 当前 | 未来 |
|------|------|------|
| 交互轮次 | 一句话→甘雨拆解 | 甘雨主动追问→用户确认→拆解 |
| 澄清方式 | 无结构化机制 | 澄清模板（scope边界/验收条件/优先级/约束） |
| 产出 | 裸 TaskNode 列表 | 已确认意图包（IntentPackage） |
| 用户参与 | 事前一句话+事后裁决 | 全程交互：事前澄清→拆解确认→中间修正 |

### 4.2 IntentPackage 类型定义

```typescript
// @cortex/shared 中预埋（Core-2a 阶段激活）
interface IntentPackage {
  id: string;
  originalIntent: string;          // 用户原始输入（保留原文，不可篡改）
  clarifiedScope: string;          // 澄清后的范围——甘雨提炼的用户真实意图
  acceptanceCriteria: string[];    // 验收条件（甘雨提炼，用户逐条确认）
  priority: "routine" | "important" | "critical";
  constraints: IntentConstraint[]; // 硬约束
  tasks: IntentTask[];             // 拆解后的任务簇
  tags: Tag[];                     // 面向 Scheduler 匹配的标签
  clarificationHistory: ClarificationRound[]; // 多轮澄清记录
  status: "draft" | "confirmed" | "dispatched" | "completed" | "cancelled";
}

interface ClarificationRound {
  round: number;                   // 第几轮（1-3）
  question: string;                // 甘雨的追问
  userAnswer: string;              // 用户的回答
  scopeChange?: string;            // 本轮澄清导致的 scope 变更
}

interface IntentConstraint {
  type: "scope" | "deadline" | "budget" | "quality" | "custom";
  value: string;
}

interface IntentTask {
  id: string;
  description: string;
  tags: Tag[];
  needsMultiPerspective: boolean;
  dependencies: string[];          // 依赖的其他 task id（在 intent 包内的 id）
  reasoningEffort: "high" | "max";
}
```

### 4.3 甘雨的澄清职责转变

**当前**：静默拆解，不做追问。

**未来**：甘雨在拆解前先进入"澄清模式"——

1. **歧义识别**：扫描用户输入，识别模糊点
   - scope 不明："优化性能"——优化什么？多快算优化？
   - 验收标准缺："做个登录页"——什么样的登录页？有哪些字段？
   - 优先级存疑：用户同时提了多个需求，哪个先做？
   - 约束缺失：有没有期限？有没有不能改的文件？

2. **结构化追问**：生成最多 3 个追问（防疲劳），按重要性排序
   - 每个追问必须具体——不问"你想要什么"，问"登录页需要包含邮箱+密码还是手机号+验证码？"
   - 追问附 default 选项，允许用户直接选："默认按方案A，除非你说B"

3. **用户回答 → 更新理解 → 确认消除**：
   - 用户回答后，甘雨提炼更新后的 scope 和 acceptance criteria
   - 向用户确认："我理解你要的是 X，验收标准是 Y，对吗？"
   - 用户确认 → 退出澄清模式，进入拆解

4. **生成 IntentPackage**：将澄清记录 + 确认后的 scope + 验收标准 + 拆解结果打包

**规则**：
- 澄清轮次上限 3 轮。3 轮后仍有歧义 → 标注 `scope=best-effort`，在 clarificationHistory 末尾记录未解决歧义，发布后允许后续修正。
- 简单意图（单标签、单一 Agent 可完成的指令）可跳过澄清——甘雨判断意图复杂度，简单意图直接拆解。

**实现方式**：纯 prompt 工程——修改甘雨的 PLANNING_SYSTEM，在拆解前加入"澄清模式"触发条件。不涉及任何架构改动。

## 五、第二层：标签匹配增强（Core-2c，Scheduler 内部增强）

### 5.1 核心思路

当前 Scheduler 的 `_findMatchingAgent` 是纯标签交集匹配——所有匹配的 Agent 平等对待。标签匹配增强在现有匹配基础上注入 TrustModel 权重和负载信息，使 Scheduler 能按"谁更适合"排序后选择最优 Agent。

**这不是市场，是排序器**——Scheduler 仍然是唯一的调度者，但它的匹配逻辑从"找到能做的"升级为"找到最适合的"。

### 5.2 匹配增强三因子

| 因子 | 信号来源 | 作用 | 权重 |
|------|---------|------|------|
| **标签匹配度** | 静态（AGENT_TAGS） | 硬门槛——不匹配的 Agent 直接排除 | 门槛（不参与排序） |
| **信任权重** | Core-2 TrustModel | 排序因子——同一能力区间内信任分高者优先 | 0.6 |
| **负载系数** | 运行时（TaskBoard.claimedBy） | 防过载——已认领超过阈值的 Agent 降权 | 0.4 |

### 5.3 匹配流程

```
IntentPackage.dispatch
  → Scheduler 遍历 tasks
  → 对每个 task：
      1. 标签交集匹配 → 候选 Agent 池
      2. 排除已达负载上限的 Agent（负载阈值：4 个进行中任务）
      3. 按 trustScore × 0.6 + (1 - loadRatio) × 0.4 排序
      4. 选择最高分 Agent → claim
      5. 如果没有 TrustModel → 回退现有纯标签匹配
  → needsMultiPerspective 节点：所有能力匹配者并行认领（不受排序限制）
```

**关键兼容性**：在 TrustModel 上线前，Scheduler 保持现有行为不变。TrustModel 上线后，通过配置开关激活匹配增强——不激活则行为不变。

## 六、第三层：自组织派发（Core-3，远期目标）

### 6.1 前提条件

自组织派发无法在当前架构中实现。以下前提缺一不可：

| # | 前提 | 当前状态 | 说明 |
|---|------|---------|------|
| 1 | **Agent-to-Agent 通信通道** | ❌ 不存在 | PipelineObserver 是单向广播（emit-only），无法形成 Agent 间双向消息回路。需要双向消息总线或事件+查询混合通道。 |
| 2 | **TrustModel 基础设施** | ❌ Core-2 预留 | 信任是自组织的基础——没有信任评分，Agent 互相不知道谁靠谱。 |
| 3 | **SkillRegistry 上线** | ❌ Core-2 预留 | Agent 自领依赖对任务性质的准确理解。SkillRegistry 提供"这个任务我以前做过"的经验基础。 |
| 4 | **Committee session 机制** | 部分存在 | 圆桌会议引擎（runMeeting）已实现自审视场景的多人协商，但未泛化为通用 Agent 间协商协议。 |

> **自组织不是市场博弈——是 Agent 基于事件的能力匹配。**没有竞标、没有出价、没有合约惩罚。Agent 看到自己能做的任务 → claim → 执行 → 报告。这是分布式的能力匹配，不是分布式市场。

### 6.2 目标形态（概念预留）

当上述前提全部满足时，自组织派发的基本形态：

1. **事件驱动 claim**：IntentPackage.dispatch → PipelineObserver 广播 `task.available` + tags → 匹配的 Agent 收到事件 → 自主调用 `TaskBoard.claim()`
2. **圆桌协商时序**：多 Agent 依赖链 → Committee session 轻量协商 → 确定执行顺序
3. **事件驱动接力**：Agent 完成 → `task.completed` 事件 → 下游 Agent 自唤醒
4. **Scheduler 降级为守护者**：不做派发——只做死锁检测、超时告警、孤儿任务回收

## 七、与现有架构的兼容路径

| 阶段 | 激活内容 | 与现有架构的关系 | 前提 | 预计投入 |
|------|---------|----------------|------|---------|
| **Core-2a** | 第一层：意图澄清对话 | 甘雨 PLANNING_SYSTEM 增加澄清模板。产出仍走当前 TaskBoard 管道。 | 甘雨 prompt 增强（纯 prompt 工程） | 极低 |
| **Core-2b** | IntentPackage 类型定义 | shared 包预埋类型。甘雨可配置走"澄清"或"直接拆解"。 | shared 包类型扩展 | 低 |
| **Core-2c** | 第二层：标签匹配增强 | Scheduler._findMatchingAgent 注入 TrustModel 排序。TrustModel 未上线则回退。 | TrustModel 上线 | 中 |
| **Core-3** | 第三层：自组织派发 | Agent 自主 claim 替代 Scheduler 集中派发。Scheduler 降级为守护者。 | Core-2 全部组件 + Agent-to-Agent 通信通道 | 高 |

> **核心兼容策略**：每阶段都保持双轨——新机制在特定条件下激活，旧机制作为兜底。不对现有路径做 breaking change。验证充分后再切轨。

## 八、与现有概念的映射

| 现有概念 | 在渐进模型中的位置 |
|------|------|
| **甘雨（战术中枢）** | 意图澄清者 + 拆解者。澄清是新增职责，拆解是现有职责。 |
| **钟离（战略把关）** | 对 IntentPackage 进行战略审查——方向对不对、契约有没有破坏。在甘雨 dispatch 前介入（optional gate）。 |
| **TaskBoard** | 保持现有角色——任务登记和认领。claim() 仍由 Scheduler 或 Agent 调用。 |
| **Scheduler** | 从纯标签匹配升级为信任+负载排序。Core-3 降级为守护者（死锁检测/孤儿回收）。 |
| **AGENT_TAGS** | 匹配的硬门槛——不变。 |
| **TrustModel** | Core-2c 注入 Scheduler 作为排序权重。Core-3 作为 Agent 互信基础。 |
| **圆桌协商** | 现有 Committee 机制泛化为 Agent 间时序协商协议。 |
| **needsMultiPerspective** | 不受排序限制——所有能力匹配者并行认领。 |
| **PipelineObserver** | 事件总线。Core-3 中 task.available / task.completed 等事件驱动 Agent 自组织。 |
| **ButlerAgent** | 澄清阶段交互 + 产出聚合呈现。 |
| **SkillRegistry** | Core-2 上线后 MetaAgent 规划时查询匹配技能。Core-3 中 Agent 自领时参考经验。 |

## 九、设计边界：不做什么

1. **不设计 Agent 间经济系统**：不引入竞标、出价、Token 结算、合约惩罚。Agent 之间是能力互补协作，不是市场交易。
2. **不修改现有 Agent 行为**：甘雨的澄清模式、Scheduler 的匹配增强——均为未来新增路径，不动现有代码路径。
3. **不定义量化参数**：信任权重系数、负载阈值——留待 TrustModel 上线后根据实测数据确定。
4. **不引入外部 Agent 接入**：Cortex 是个人工具链，不存在多租户、外部 Agent 注册、跨项目任务流通。
5. **不在当前阶段追求 Vibe Coding 模式**：Agent 间自由协作需要双向通信通道——这是 Core-3+ 的范畴。
