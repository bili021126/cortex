### 1.2 断开的链路——为什么 solo-flight 冷启动后技能=0

```
                    [产出文件层]  ───── 断开 ─────  [记忆持久化层]
  analysis → design.md          SkillExtractor ← LoopAgent only
  code → src/*.ts                ↓                      ↓
  review → review.md          SkillRegistry (内存) → MemoryStore? ❌ 从未写入
  loop → pattern.md               ↓
  inspector → 侦察报告      下次冷启动 → 加载? ❌ 从未从 DB 加载
```

四条致命断裂：

1. **提取范围过窄** — `_extractAndRegisterSkills` 只对 LoopAgent 的输出触发（`scheduler.ts:586`）。但是 MetaAgent 规划的任务图里 LoopAgent 不是必选项——solo-flight 就没有 LoopAgent。
2. **注册表是内存孤岛** — 技能写入 `SkillRegistry`（内存 Map），但从未调用 `memory.writePending()+commitMemory()` 写入 MemoryStore。重启即清零。
3. **冷启动不会加载** — `SkillRegistry.registerAll()` 注释写着"用于从 MemoryStore 的 Skill 类型记忆中批量加载"，但整个代码库没有任何地方调用它。
4. **文件产出无回溯机制** — pattern.md / design.md / review.md 已经落盘，但没有机制事后扫描这些文件提取技能。

## 二、闭环设计

### 2.1 总览

```
  ┌─────────┐    输出文件     ┌──────────────────┐
  │  Agent  │ ──────────────→ │ pattern.md        │
  │  执行   │                 │ design.md         │
  └────┬────┘                 │ review.md         │
       │                      └────────┬─────────┘
       │ ① 每节点完成后              │
       ▼                              ▼
  ┌─────────────────┐    ┌──────────────────────┐
  │ SkillExtractor   │    │ FileSkillScanner     │ ← ② 文件回溯扫描
  │ (实时提取)       │    │ (事后批处理)         │
  └────────┬────────┘    └──────────┬───────────┘
           │                        │
           ▼                        ▼
  ┌─────────────────────────────────────────────┐
  │            SkillRegistry (内存)              │
  │          tag/agent/id 三索引                 │
  └─────────────────────┬───────────────────────┘
                        │
               ③ writePending + commitMemory
                        │  (MemoryType.Skill, subType:Fact)
                        ▼
  ┌─────────────────────────────────────────────┐
  │              MemoryStore (SQLite)            │
  │           .cortex/memory-*.db                │
  └─────────────────────┬───────────────────────┘
                        │
             ④ cold-start 加载
                        │  read({ memoryType: "SKILL", state: "ACTIVE" })
                        ▼
  ┌─────────────────────────────────────────────┐
  │         SkillRegistry.registerAll()          │
  │         → MetaAgent.queryByTags()            │
  └─────────────────────────────────────────────┘
```

一条完整的技能生命线：
- **① 实时线**：每个 Agent 节点执行完毕 → `pipeline._rememberResult` → 从 output 提取 SkillTemplate → 写入 MemoryStore
- **② 回溯线**：冷启动后 / 合并大考后 → 扫描已产出文件（pattern.md, design.md）→ 提取遗漏的技能
- **③ 持久线**：SkillRegistry 每次 register 后 → `memory.writePending()` + `memory.commitMemory()` 落 SQLite
- **④ 加载线**：`MemoryStore.init()` 后 → `read({memoryType: "SKILL", state: "ACTIVE"})` → `SkillRegistry.registerAll()`

### 2.3 核心数据流

```
  Agent 产出 output 字符串
       │
       ▼
  scheduler._onTaskComplete(node, result)
       │
       ▼
  pipeline._rememberResult(node, result)
       │  ┌── 现有逻辑：writePending + commitMemory (Episodic, subType:Fact)
       │  │
       │  └── 新增逻辑 ──────────────────────┐
       │                                      ▼
       │                          extractSkillsFromOutput(output)
       │                                      │
       │                          ┌───────────┴───────────┐
       │                          │ skills.length > 0?     │
       │                          └───────────┬───────────┘
       │                              Y       │       N
       │                                      │       → return（无技能则跳过）
       │                                      ▼
       │                          for skill of skills:
       │                              skillRegistry.register(skill)
       │                              memory.writePending({
       │                                  memoryType: MemoryType.Skill,
       │                                  subType: MemorySubType.Skill,
       │                                  content: skill as Record<string,unknown>,
       │                                  summary: `[技能沉淀] ${skill.agentType}:${skill.name}`,
       │                                  agentType: AT.Loop,
       │                                  creatorId: "skill-extractor",
       │                              })
       │                              memory.commitMemory(skillMemId)
```

### 2.4 冷启动加载流程

在 `MemoryStore.init()` 后、`Scheduler` 创建前：

```typescript
// 从记忆库加载已沉淀技能
const skillEntries = memory.read({
  memoryType: MemoryType.Skill,
  state: MemoryState.Active,
  queryMode: "csa",
  limit: 100,
});

const skillTemplates = skillEntries
  .filter(e => e.content && typeof e.content === "object")
  .map(e => e.content as unknown as SkillTemplate)
  .filter(t => t.id && t.agentType && t.triggerTags && t.steps?.length > 0);

skillRegistry.registerAll(skillTemplates);
```

## 三、SkillExtractor 增强——Markdown 模式提取

现有的 `extractSkillsFromOutput` 只能从 JSON 块提取。但 Agent 的产物大多是 Markdown（design.md、pattern.md、review.md），需要增加非结构化提取能力。

### 3.1 新增功能：从 Markdown 提取模式

```typescript
function extractSkillsFromMarkdown(content: string): SkillTemplate[] {
  // 策略1: 提取 "## 模式 X: ..." 段落
  //   → 段落标题 → name
  //   → "适用场景: ..." → trigger
  //   → 编号列表项 → steps
  //   → 段落首句 → expectedOutput
  
  // 策略2: 提取 "## 可复用技能 / 可沉淀模式" 章节
  //   → 每个子标题是一个技能
  
  // 策略3: 回退——整个内容作为一个技能模板

  // 每个提取的技能:

  //   - status: "trial"

}
```

## 五、文件回溯扫描——为合并大考准备

合并大考场景特殊：
- 冷启动 SkillRegistry 为空
- 但 archive 目录下有上一轮的 pattern.md / design.md / review.md
- 应该在合并开始前扫描这些文件，提取技能注入 SkillRegistry

合并大考流程增强：
```
  1. MemoryStore.init(memory-merge-exam.db)
  2. 加载上一轮记忆中的 SKILL 类型（若有）
  3. scanOutputFilesForSkills(archiveDir) → 提取更多技能
  4. skillRegistry.registerAll(...)
  5. MetaAgent(skillRegistry) → 规划时能感知上一轮的发现
  6. Scheduler(skillRegistry) → 每个节点完成后提取技能 + 持久化
```

## 七、不做什么（边界）

- **不做 SkillExecutor**：技能的执行引擎（解析 steps 并自动调用工具）留到 Core-3
- **不做 adoption/rejection 自动升降级**：adoptionCount/rejectionCount 由后续 LoopAgent 审视时才更新
- **不做跨项目技能共享**：每个项目独立 .cortex/memory-*.db，技能不跨项目迁移
- **不改造 LoopAgent**：LoopAgent 现有 prompt 已经正确，只是没有被调用和执行链路
