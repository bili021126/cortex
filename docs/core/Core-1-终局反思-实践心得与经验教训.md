# Core-1 终局反思——实践心得与经验教训

> Core-1 完整阶段复盘。覆盖协议约束、调度演进、Agent 角色化、记忆系统、测试验证、圆桌对话预演六大主线。
> 归档于 `docs/core/`，作为 Core-2 启动前的工程基线。

---

## 一、Core-1 的实际轨迹

Core-1 不是按蓝图一次性交付的。实际演进是：

```
协议约束地基 → 调度闭环 → 重规划机制 → E2E 真实 LLM 链路
    → 测试暴露标签匹配缺陷 → _findMatchingAgent 重构
    → Agent 角色化（原神人格）→ 调度 bug 被角色化输出肉眼发现
    → 圆桌对话预演 → MemoryStore Episodic/Knowledge 双轨验证
    → 识别物理形态瓶颈 → Committee 推迟
    → 过渡阶段债清点 → 宣布收尾
```

每一步前一步踩出的坑推动了后一步的修正。没有一步是"按计划来的"，但每一步都学到了设计文档里写不出来的东西。

## 二、工程实践的 8 条心得

### 1. 协议约束比功能先行更重要

Core-1 早期最正确的决策不是做了哪个 Agent，而是把散落各处的运行时契约集中为可验证的协议：

- `peek()` 替换 `get()`，返回冻结副本——堵住了绕过 CAS 直改状态的缺口
- `release(nodeId, agentType)` 原语——彻底消灭了 `claimed` 节点无人释放的死锁
- 所有 Agent 删除 `setGate/setLockManager`——去掉了手动注入的耦合点

C1-C3 临界缺陷（claim 与 spawn 之间的单向承诺缺口）爆发时，正是这套协议框架兜住了修复——只需要在缺口中补一段 complete(false) 自愈逻辑，不需要重构整个调度链路。**协议不是锦上添花，是止损的防火墙。**

### 2. 测试是设计发现工具，不是验收工具

`_findMatchingAgent` 从"首个匹配"改到"最佳匹配 + 平局加分"，不是因为设计文档说该这么做，而是因为测试跑出了"Review 节点被 Code 执行"。TAG_VOCABULARY 补全 inspect/doc_govern、AGENT_TAGS 权重对齐，全是测试驱动的。

Inspector 的 MAX_LOOPS=5 跑不完数据采集，硬编码的限制在真实数据面前暴露了无知。**硬编码的上限本质是在假装知道边界条件。**

### 3. 角色化不是装饰，是可调试性基础设施

最初只是一句"agent 像流水线工人"，但后果远超预期：

- **肉眼发现调度 bug**：角色化输出让你一眼看到"刻晴（review）的任务怎么是阿贝多（code）在执行"，比任何断言都直观
- **催生了新协作模型**：圆桌对话从流水线任务分配变成了角色自主发言 + PASS 机制——这不是任务系统，这是议会
- **验证了记忆双轨**：Episodic（对话历史）+ Knowledge（世界观背景），正好是 Committee 机制的记忆基础

教训：**好的可视化（哪怕只是 emoji + 名字）比纯结构化日志更早暴露调度和逻辑问题。** 这不是美学，是工程。

### 4. 架构演化靠摩擦驱动，不是蓝图驱动

| 组件 | 最初形态 | 摩擦 | 最终形态 |
|------|---------|------|---------|
| Scheduler | 只有 `topologicalSort` | CortexEngine 成为上帝类 | 完整执行生命周期（executeAll + replan 动态消费）|
| _findMatchingAgent | 首个匹配 | 调度到错误 Agent | 打分制 + 平局 node.type 加分 |
| MemoryStore | CRUD + SQLite | 检索逻辑膨胀 | 正在拆出 MemoryRetriever |
| 标签体系 | 部分覆盖 | Inspector/DocGovern 调度失败 | TAG_VOCABULARY + AGENT_TAGS 全量对齐 |

规律：代码不会因为"设计文档说该这么做"而变好，只会因为"不改这里就天天踩坑"而变好。

### 5. 阶段之间必须有还债缓冲区

从 Nano 到 Meso-Lite 到 Core-1，每次冲完新功能后如果没有过渡阶段还债，下一阶段的代码会直接建在松动的地基上。Core-1 冲完后的债（MemoryStore 检索 AND 过严、SQLite 连接管理、LLM 文件缓存、E2E 不可重复）如果直接带入 Core-2，Committee 的上下文检索、裁决记录落盘、治理流程调试全都会踩坑。

**过渡阶段只做巩固，不加新功能。这是阶段模型的宪法级约束。**

### 6. 物理形态不确定，治理层就是空中楼阁

Committee 三级收束协议的设计是清晰的——但 CLI 下只能打印文本 + 等待 stdin `y/N`，根本无法承载委员会审议→裁决→处分的闭环。这不是功能没做完，是交互载体没到位。

教训：对于依赖用户交互的架构层，**必须先明确物理形态（或至少抽象出统一接口，如 UserJudgementGateway）**，否则就会陷入"有眼无手"——能看（PipelineObserver、DocGovern 报告），不能动（审议、裁决、处分）。

### 7. 区分架构债和运维债

| 类型 | 例子 | 处理方式 |
|------|------|---------|
| 架构债（阻塞下一阶段） | 关键词 AND 过严、SQLite 连接管理 | 过渡阶段必须还 |
| 运维债（影响可靠性但不阻塞设计） | LLM 重试熔断、recording/replay | Core-2 或 Full 阶段再做 |
| 测试摩擦力（影响效率但不阻塞正确性） | E2E 无文件缓存 | 过渡阶段做轻量级缓解 |

不是所有"跑起来不爽"的东西都是债。区分标准：**不还它会不会让下一阶段的代码写出来就是错的。**

### 8. 圆桌对话是一个意外发现的新原语

conversation-10.ts 证明了多 Agent 之间可以有比"任务派发→执行→汇报"更丰富的协作模式。串行发言、基于共享记忆自主决定是否说话、世界观背景统一语境——这个模型天然适配 Committee 机制。

**建议 Core-2 把这个模型从测试脚本升级为 engine 层的抽象（如 RoundTableSession / CommitteeSession）。**

## 三、正判与误判

### 正判

- **协议约束先行**：CAS 封闭 + release 原语 + 构造注入，在 C1-C3 临界缺陷爆发时兜住了修复成本
- **双轨测试**：Mock 验证管线 + E2E 验证大脑，分工清晰
- **Committee 推迟**：不是因为懒惰，是因为正确识别了物理形态瓶颈
- **角色化作为可调试性投资**：投入小，回报远超预期

### 误判

- **标签体系未在早期穷举**：TAG_VOCABULARY 和 AGENT_TAGS 的缺口是调度问题的根源，应该在 Agent 注册时自动校验
- **BFS 检索在 Core-1 过早投入**：关联边密度不足时 BFS 是空架子，真正该优先的是 CAS 状态机
- **LLM 缓存只做了内存级**：一个十几行的文件缓存就能省掉 90% 的重复 API 费用和网络超时焦虑

### 规律

**每一轮冲完之后，都有一个自然的"看清楚了"时刻。** 不是因为你更聪明了，是因为错误已经替你交过学费了。过渡阶段的存在意义就是把这个学费变现——把"踩过的坑"转化为"下一轮不会踩的约束"。

## 六、Core-2 的三重前置条件

Core-2 的治理层需要三个前提同时就位，缺任何一个都会让 Committee 变成空中楼阁：

| 前置条件 | 解决的问题 | 缺失的后果 |
|---------|-----------|-----------|
| **Electron 原型** | 人能看报告、做裁决、拍板 | 治理闭环不成立——DocGovern 产出报告没人看 |
| **MCP 集成** | Agent 能碰到真实的文件和系统 | Agent 是活在模拟世界的宠物——审议再严谨，决策再高明，操作的都是假数据 |
| **Committee MVP** | 审议→裁决→处分的治理回路 | 有眼有手但没有决策机制 |

**MCP 是 Core-2 的准入条件，不是可选功能。**

Core-1 的所有 Agent 执行本质上都是对假想场景的推演——它们能"说"自己会做什么，但不能真正操作文件、执行命令、访问外部系统。没有 MCP，Core-2 的治理层就是在治理一个模拟世界，委员会裁决的对象是虚构的。

这也是 Core-2 一再推迟的深层原因：**Electron + MCP + Committee 是治理层的铁三角，任何一个没就位，写了 Committee 代码也是空跑。** 这不是拖延，是守住底线。

## 七、从 Core-1 到 Core-2 的经验传递

1. **协议约束是地基**——没有它，C1-C3 那种临界缺陷会成片爆发
2. **代理不是越多越好**——8 个 Agent 已经覆盖了当前所有用例，Core-2 不应盲目扩张
3. **角色化值得保留**——不是玩具，是调试、测试、委员会交互的统一载体
4. **测试不可重复不是小问题**——LLM 文件缓存是 Core-2 治理层调试的最低门槛
5. **物理形态 + MCP 双前置**——Electron 让人能裁决，MCP 让 Agent 能实操。两条腿都站稳了，治理层代码才有验证的土壤
