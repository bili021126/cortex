> 定位：v2.0 宪法颁布后，围绕委员会重定位、双轴治理模型、隐喻体系补全展开的设计推演记录。此文为讨论存档——不修宪，不纳入当前阶段实现规划。
> 时间：2026-05-07

## 一、委员会重定位——按领域切分，不按阶段切分

### 1.1 旧模型的问题

v1.1/v2.0 早期定义中，常设委员会与临时委员会之间是"诊断→决策"的串行关系：
- 常设委员会做问题诊断（归因与范围界定）
- 临时委员会在诊断完成后按议题召集决策

这个 handoff 边界模糊——诊断天然包含判断，"归因到安全问题"本身已是一种决策。

### 1.2 新切分：治理审计 vs 执行裁决

按**领域**切分，两者成为互不依赖的两条独立线：

| | 常设委员会 | 临时委员会 |
|---|---|---|
| **领域** | 治理审计（Governance Audit） | 执行裁决（Execution Arbitration） |
| **触发** | 持续巡逻，主动发现 | 事件驱动，按需召集 |
| **输出** | 审计报告、修宪建议、技术债清单 | go/no-go 决定、升级/降级判断、调度调整 |
| **Owner** | DocGovernAgent 主导 | MetaAgent 主持 | 召集 |
| **用户参与** | 大事必须在场，小事可不参与 | 按四层上报机制决定是否到场 |

### 1.3 成员构成

**常设委员会**（治理审计）：
- DocGovernAgent（常任审计，主导）
- 你（常任最终决策）
- ReviewAgent / AnalysisAgent（按需）
- MetaAgent **不入常设委员会**——治理不是它的领域

**临时委员会**（执行裁决）：
- MetaAgent（主持召集）
- DocGovernAgent（顾问视角列席）
- 相关 Agent（按议题）
- 你（按上报层级决定）

### 1.4 旧"诊断"职能的归宿

原来"常务委员会诊断职能"中"诊断"的实际内涵被拆解到两个目的地：
- **治理诊断**（发现文档不一致、接口泄露、宪法咬合问题）→ 常设委员会的审计闭环
- **执行诊断**（Agent 失败归因、冲突定性、风险等级判断）→ MetaAgent 在临时委员会前自行判断，不需要常设委员会前置

## 二、"事至上而下，权自下而上"——双轴治理模型

### 2.1 双轴定义

- **事轴（命令流）**：用户意图 → MetaAgent 规划 → 任务树分解 → Agent 执行。命令往下走。
- **权轴（约束流）**：Agent 发现风险 → 上报升级 → 临时委员会裁决 → 最终到用户。约束往上走。

### 2.2 双轴体系中的组件定位

```
事至上而下（执行流）
  用户
   │
   ▼
  MetaAgent ──→ 任务树 ──→ Agent 执行
   │                           │
   │                           │ 风险/冲突/失败
   │                           ▼
   │                    临时委员会（执行裁决）
   │                           │
   │                           ▼
   ◄──────────── 你（最终裁决）──┘
                权自下而上（约束流）


横切监督（独立于双轴之外）
  PipelineObserver ──→ DocGovernAgent ──→ 常设委员会 ──→ 审计报告 → 你
                                              │
                                         TrustModel（记分）
                                         ConfirmGate（阻断）
```

### 2.3 MetaAgent 的双轴跨越

MetaAgent 是架构中**唯一跨越双轴**的实体：
- **事轴**：命令流枢纽——用户意图进来，它规划、分解、派发
- **权轴**：临时委员会召集人——Agent 底部上报的风险由它主持裁决

但跨越不等于拥有。两条轴的主权都不在 MetaAgent：
- 事轴的源头是用户，不是它
- 权轴的最终裁决（Level 4）是你，不是它
- 它完全不碰治理审计（常设委员会对它来说是外部约束）

核心表述：**联系最多、覆盖面最广、但权力边界最硬**——执行归它，主权归你。

## 三、治理层延伸——更新后的五组件链

### 3.1 旧五组件

治理层不独立成层，作为横切关注点嵌入五个现有组件：
DocGovernAgent、ConfirmGate、Committee、PipelineObserver、TrustModel

### 3.2 委员会拆分后的新图

临时委员会从治理延伸路径中退出——它是执行裁决工具，不是治理组件。

更新后的治理延伸五组件：

```
DocGovernAgent（审计引擎）
       │
       ▼
常设委员会（议事场——审议审计发现，形成修宪建议）
       │
       ▼
ConfirmGate（执行约束——不可绕过）
       │
       ▼
PipelineObserver（持续监测——只发射信号，不干预执行）
       │
       ▼
TrustModel（信任记分——形成免疫记忆）
```

### 3.3 技能治理缺口

此前讨论确认的技能治理缺口天然落在常设委员会的治理审计议题中：
- 技能注册必须经 DocGovernAgent 独立审查
- 试用→生产门禁
- 废弃检测
- 跨 Agent 权限继承冲突解决

## 四、隐喻体系——纪检委补全监督流

### 4.1 三套隐喻体系的分工

| 隐喻体系 | 覆盖域 | 核心组件映射 |
|---------|--------|------------|
| **政府隐喻** | 执行流 | MetaAgent（执行皇帝）、Agent Pool、任务树 |
| **纪检隐喻** | 监督流 | PipelineObserver、DocGovernAgent、常设委员会 |
| **议会隐喻** | 议事形式 | 常设委员会（专门委员会）、临时委员会（特别委员会） |

### 4.2 纪检委映射

纪检委隐喻在政治隐喻体系内自洽，不需要跳出到生物隐喻（免疫系统）：

| 纪检职能 | Cortex 映射 | 行为 |
|---------|------------|------|
| 巡视组 | PipelineObserver | 到处转，看出问题只能写报告（emit），不能当场抓人 |
| 立案审查 | DocGovernAgent | 收到巡视线索，启动正式程序，调档案、查日志、形成证据链 |
| 审理室 | 常设委员会 | 拿证据链开会审议，出处分建议 |
| 处分落地 | TrustModel 降分 / ConfirmGate 加锁 / 修宪建议 | 制裁不在纪检委手里执行，由对应组件实施 |
| 最终决定权 | 你（用户） | 纪委只有建议权，批不批你说了算 |

### 4.3 这个隐喻的独特解释力

解释了架构中"看"与"拦"的分层：
- PipelineObserver 只能 observe→emit，不能 intervene——巡视组只能写督察报告
- DocGovernAgent 能启动审查但不能执行惩罚——立案权≠处分权
- ConfirmGate 能拦截是因为它是"物理关卡"——不是审查者，是关卡本身
- TrustModel 降分是自动机制——类似记过自动影响晋升，不需要每次开会

临时委员会**不在纪检体系里**——它不是查问题的，是前线作战会议。

## 五、四层错误逐级上报

Level 0～4 全是"权自下而上"的形式化通道：

| 层级 | 机制 | 含义 |
|------|------|------|
| Level 0 | Agent 自愈 | 内部重试、降级策略 |
| Level 1 | 同级协商 | Committee 内其他 Agent 协助 |
| Level 2 | 重新调度 | MetaAgent 换 Agent 重新派发 |
| Level 3 | 重规划 | MetaAgent 修改任务树 |
| Level 4 | 用户裁决 | 反馈到你，你决定 |

全部通过返回值驱动，不经过事件总线。

## 六、民主集中制的位移

模式没有消失，"集中"的中心从 MetaAgent 转移到了你（用户）。

三层递进裁决：
1. **事实最高**：技术事实由领域专家提供，不接受外行投票否决
2. **基线优先**：凡有宪法/契约/历史先例可循，依基线自动裁决，不上升
3. **分歧交付**：只有真正无法依规则裁决的价值判断才上升到用户

这意味着经过前两层过滤后，到你手里的已经是真正需要判断的事——不是甩锅。

## 七、PipelineObserver 的退化防线

当前 API 物理形态提供了天然防御：
- `emit()` → `handler(event)`，handler 返回 void，emit 丢弃返回值
- 不存在 `emitAndWait()` 或 request-response 模式
- 单向广播，无法形成控制回路

退化风险来自**未来可能的 API 扩展**——一旦有人加了双向模式，防线就破了。当前不需要代码层面加固，但需在 API 文档/契约中明确标注"单向不可回"约束。

---

## 八、任务树上的 contextLevel

在 TaskNode 上新增 `contextLevel: 1 | 2 | 3` 字段，控制 Agent 看到多少上下文：

| Level | 可见范围 | 对齐 v1.1 概念 |
|-------|---------|-------------|
| 1 | 仅自身节点 | 信息熵最低——盲目执行 |
| 2 | 上游结果 | 知道前置发生了什么 |
| 3 | 完整树 + 兄弟节点 + 重规划历史 | 信息熵最高——全局视野 |

## 九、TrustAgent 的角色扩展

TrustModel 拆分为：
- **TrustModel**（确定性记分簿，基础设施）——自动衰减、事件驱动的分数变更
- **TrustAgent**（Agent Pool 中的 LLM 评估者，只读工具）——深度行为评估

TrustAgent 承担两项职责：
1. **考评**：对 Agent 行为和产出做定性评估，不只看 TrustModel 数字
2. **铨选**：调度时推荐最优 Agent（基于历史表现 + 当前任务匹配度），由 MetaAgent 采纳或驳回

## 十、Core-1 反思轮——Agent 体系扩展与治理补全（2026-05-04）

### 10.1 背景

Core-1 第三轮交付后，Agent 池仍是统一的 AgentRunner 套 6 套提示词。反思轮聚焦三个问题：Agent 是否该独立实现、工具权限该在哪、治理层是否需要延伸。

### 10.2 Agent 池从 6 扩至 12

```
原有 6: Code, Review, Analysis, Ops, Loop, DocGovern
新增 6: Butler, Inspector, Api, Browser, Data, MetaAgent（中枢定位明确化）

Core-1 落地 8: 6 原有 + Butler + Inspector（Api/Browser/Data 留 Core-2）
```

关键决策：按行为模式拆分独立类，而非通用 AgentRunner 换提示词。共用 ReAct 循环的（Code/Review/Analysis/Ops/Loop）提取 react-helper.ts，DocGovernAgent 和 InspectorAgent 走独立管线。

### 10.3 工具权限从 Agent 移到 Toolkit

此前宪法原则三规定"Agent 的 allowedTools 是唯一权限边界"。反思轮发现该设计导致：每个 Agent 类各自维护工具白名单（重复），Agent 和权限耦合（换工具要改 Agent 代码）。

修正：`AGENT_TOOL_PERMISSIONS` 集中在 shared/types 层，Toolkit.execute() 按 callerType 查表拦截。Agent 只传身份，不持有权限定义。v2.2 宪法已修。

### 10.4 Agent 状态机

原宪法 5.2 只有"创建→认领→执行→销毁"四阶段，无唤醒概念。反思轮引入五态模型：

```
Created → Awake → Active → Awake → ... → Draining → Destroyed
```

只有 Awake 态可接收 execute()。常驻（Butler/MetaAgent）始终 Awake，其余按需唤醒。

### 10.5 治理层新组件

原治理层只有 DocGovernAgent + 常设委员会 + 临时委员会。反思轮补充四个组件：

| 组件 | 对应委员会 | 职责 | 归属 |
|------|----------|------|------|
| IncidentEscalator | 都察院 | 故障归因+路由决策。N 轮失败后判定是代码问题、Agent 问题还是环境问题 | Core-2 |
| ContractEnforcer | 给事中 | 部署前契约校验。API schema/DB migration 兼容性 | Core-2 |
| CircuitBreaker | 大理寺 | N 次同因失败→熔断→标记 node.blocked | Core-2 |
| InspectorAgent | 独立 Agent | 纯粹事实输出：依赖图谱、耦合度、影响面。LLM 仅格式化，不做推理补充 | Core-1 |

### 10.6 InspectorAgent——事实基础层

InpectorAgent 不是分析者、不是审查者、不是治理者。其输出是所有 Agent 和用户决策的**信息基础**：

- 输入：代码库
- 行为：tsc --noEmit → madge 依赖分析 → AST walker 扫接口签名 → grep 调用方
- 输出：结构化事实（"userService 被 14 个文件引用"），无评价、无建议
- LLM 仅做格式化，不做推理补充——这是可靠性的保证

Inspector → ButlerAgent → 用户 → 决策归档 → DocGovernAgent。形成"事实→决策→约束"闭环。

### 10.7 MetaAgent 职责瘦身

此前 MetaAgent 负载过重：规划+重规划+Committee 召集+测试场景骨架。反思轮将：
- Committee 召集 → 归 Scheduler
- 测试场景骨架 → 归 AnalysisAgent
- MetaAgent 保留：意图→TaskBoard 规划 + requestReplan。

### 10.8 ButlerAgent 角色明确化

ButlerAgent 从"独立于 Engine 的管家进程"正式纳入 Agent 池（v2.2 修宪）。五大法定职责：
1. 唯一交互出口
2. 决策中转（Inspector→用户→归档）
3. 闲时采集（闲聊中自然收集决策原因，不显式追问）
4. 事件通知
5. 入口适配（无项目→全屏对话；有项目→IDE 三栏）

### 10.9 测试分层

- **代码层**（冒烟）：ApiAgent/BrowserAgent 内嵌执行——打所有端点、加所有页面。通了才进场景层。失败→打回 CodeAgent。
- **场景层**（业务流程）：MetaAgent 给骨架（"验证登录成功"）→ AnalysisAgent 补细节（找到对应 selector）→ BrowserAgent 执行。

### 10.10 架构决策归属

复杂架构决策（拆不拆微服务、选什么协议）Agent 只提供分析+选项，不做决定。用户决策后归档为约束依据。InspectorAgent 提供事实依据，ButlerAgent 转译，用户裁决，DocGovernAgent 归档。

### 10.11 界面形态

- 顶部状态栏：Agent 圆点颜色语义（绿=干活、蓝=常驻、白=闲、灰=睡、红=异常）
- 双入口：无项目→ButlerAgent 全屏对话；打开项目→自动切换 IDE 三栏（文件树+编辑器+对话面板）
- 隐喻可视化：宪法中的"三省六部"→顶栏灯就是"朝堂"

### 10.12 闲置行为

Agent 闲时可以出来闲聊（非任务模式）。聊到项目相关→自然采集决策原因；聊到技术→Agent 之间可以争论技术细节，用户旁听学习。争论边界：技术选择可以争，项目决策和涉及 ConfirmGate 的操作不可争，由用户裁决。
