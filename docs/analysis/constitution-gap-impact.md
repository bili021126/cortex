| 缺口 | 严重度 | Core-2 风险 | 风险等级 |
|------|--------|------------|---------|

### 核心判断

> **8 处缺口中，4 处对 Core-2 构成高或阻塞级风险。其中 2 处（3-A 审计闭环、3-C 阶段门禁）直接决定 Core-2 能否安全启动。**  
> 已生成的修复提案（AM-2026-0606-002、-003）的正确方向值得肯定，但 **代码层的配套跟进和宪法定义的完备性** 仍需在 Core-2 启动前完成。

## 二、逐项分析：缺口对 Core-2 的架构影响

---

### 🔴 发现 1-A：自反性缺口 —— 宪法层已修复，代码层硬编码锁未解

**当前状态**：✅ AM-2026-0606-001 已裁决通过，宪法 v2.5.13 新增子约束7「子约束修改规则」。  
**代码层状态**：❌ `amendment-judge.ts` 的 `CHECK_ORDER` 仍是硬编码常量数组。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **紧急修宪通道**（发现 1-B） | 子约束7 为新增子约束提供了宪法依据，但 `CHECK_ORDER` 硬编码意味着新增子约束（如紧急修宪子约束8）需要手动修改评判引擎代码——这不是自动化管线应有的行为 | 🟡 中 |
| **Core-2 治理组件入宪** | Core-2 的 IncidentEscalator、ContractEnforcer、CircuitBreaker 等治理组件需要入宪。若需新增原则或子约束，`CHECK_ORDER` 无法自动扩展 | 🟡 中 |
| **阶段跃迁修宪** | Core-1→Core-2 阶段跃迁必然涉及宪法修改（阶段限定条款、新 Agent 入宪等）。修宪提案的评判仍然依赖硬编码的 CHECK_ORDER | 🟢 低（现有检查项够用） |

#### 架构根系追踪

```
宪法层（已修复）
  原则七 子约束7「子约束修改规则」
    └── 定义了子约束修改的 5 项审查条件 (a)-(e)

代码层（未跟进）
  amendment-judge.ts
    └── CHECK_ORDER = const [...]      ← 硬编码，不可扩展
    └── evaluateAmendment()             ← 无子约束修改专项检查
    └── determineVerdict()              ← 无法区分普通提案与子约束修改提案
  governance-loop.ts
    └── loadPendingProposals()          ← 无法标记「原则七·子约束」提案
    └── judgeProposals()                ← 无子约束修改的强化审查路径
```

#### 修复建议

1. **CHECK_ORDER 可扩展化**：改为注册表模式（`Map<string, CheckFn>`），新增子约束时调用 `registerCheck()` 自动注入
2. **提案分类标记**：`loadPendingProposals()` 检测 `section` 字段含「原则七·子约束」时，标记 `isSubconstraintModification: true`
3. **强化审查路径**：`determineVerdict()` 对有标记的提案应用更严格的审查（双重把关验证）

### 🟡 发现 1-B：紧急修宪通道缺失 —— Core-2 启动期的安全隐患

**当前状态**：❌ 未修复，无提案生成。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **新 Agent 引入风险** | Core-2 将 ApiAgent、BrowserAgent、DataAgent 从预留转为落地。若某个 Agent 在落地过程中暴露安全漏洞，需要紧急修宪修改安全边界定义或 Toolkit 权限——标准流程可能太慢 | 🟡 中 |
| **Sentinel/TrustModel 入宪** | Sentinel 和 TrustModel 是 Core-2 的基础设施组件，入宪过程中若发现宪法现有安全条款与实现冲突，可能需要紧急修补 | 🟡 中 |
| **ElectronAdapter 接入** | IPC 弹窗引入新的攻击面。若发现 CLIAdapter 的路径隔离在 Electron 环境下失效，需要快速修宪加固读取安全边界（§7.5） | 🟡 中 |
| **治理组件熔断** | CircuitBreaker 的熔断阈值若在宪法中定义过严或过松，上线后可能立即触发误熔断或漏熔断——紧急调整需要快速通道 | 🟡 中 |

#### 架构依赖

```
Core-2 新组件上线流程（正常）
  设计 → 审计（DocGovernAgent）→ 修宪提案 → 评判 → 裁决 → 写入 → 验证
  （标准流程，1-B 不阻塞）

Core-2 紧急场景（1-B 缺失）
  安全漏洞发现 → 无快速通道
    → 走标准流程（可能延误数小时/天）
    → 漏洞利用窗口扩大        ← ❌ 风险
```

#### Core-2 启动建议

Core-2 启动前应至少完成紧急修宪通道的宪法定义，不一定需要代码实现完毕。这应设为 Core-2 的准入条件之一（与 3-C 阶段门禁定义联动）。

### 🟡 发现 2-A：DECISION_REQUIRED 回退机制缺失 —— Core-2 治理组件的关键依赖

**当前状态**：⏳ AM-2026-0606-003 已提案，待裁决。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **常设委员会通知** | Core-2 常设委员会产出审计报告后需通过 DECISION_REQUIRED 呈报用户。若无回退机制，用户离线时系统阻塞——委员会无法关闭审计发现 | 🟡 中 |
| **监理封驳** | 监理发现违宪操作触发封驳，需用户裁决。若用户不响应，封驳状态挂起——后续执行链全部阻塞 | 🔴 高 |
| **ContractEnforcer 契约校验** | 部署前契约校验失败触发 DECISION_REQUIRED——用户必须决定"放行还是打回"。无超时处理 = 部署管线永久挂起 | 🔴 高 |
| **CircuitBreaker 熔断恢复** | 熔断后恢复需用户确认。用户不响应 = 熔断组件永远不可用 | 🟡 中 |
| **阶段门禁裁决** | Core-1→Core-2 阶段跃迁需要用户最终裁决。DECISION_REQUIRED 回退机制缺失 = 用户无法决策时阶段跃迁卡死 | 🔴 高 |

#### 架构依赖图

```
Core-2 治理组件 → 依赖 DECISION_REQUIRED 轨道
  ├── 常设委员会：审计发现呈报
  ├── 监理实体：违宪封驳裁决
  ├── ContractEnforcer：契约冲突裁决
  ├── CircuitBreaker：熔断恢复确认
  ├── TrustAgent：铨选推荐确认
  ├── 阶段门禁：Core-1→Core-2 跃迁裁决
  └── 吏部/户部：人事任免/资源配额确认
```

**关键发现**：AM-2026-0606-003 定义的 four-layer safety valve（超时/防抖/离线降级/审计逃逸）是 Core-2 治理组件正常运行的必要前提。如果该提案不被裁决通过，Core-2 的治理组件将在不可靠的通知基础上构建——每一层依赖 DECISION_REQUIRED 的组件都面临永久阻塞风险。

### 🟡 发现 2-B：治理事件接入路径未入宪 —— 凝光的判决走不出群玉阁

**当前状态**：❌ 未修复，无提案生成。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **常设委员会运行** | 常设委员会的审计结论无法进入通知管线 = 用户不知道委员会在审什么、发现了什么、需要做什么决策 | 🔴 高 |
| **审计闭环（3-A）** | 发现 3-A 的修复（AM-2026-0606-002）定义了审计闭环流程，但闭环的「通知整改责任人」环节依赖通知管线——审计事件不入管线，闭环第一步就走不通 | 🔴 高 |
| **ButlerAgent 治理角色** | 宪法 §6.1 定义 ButlerAgent 为「唯一交互出口」。治理事件不走通知管线 = ButlerAgent 无法向用户呈报治理信息——ButlerAgent 的治理职责被架空 | 🟡 中 |
| **PipelineObserver 治理事件类型** | 当前 PipelineObserver 的事件类型（PipelineEventType）中缺少治理相关事件（如 `governance.audit`、`governance.constitution_check`）。Core-2 新增治理组件时若不加治理事件类型，所有治理结论都「写完磁盘即结束」 | 🔴 高 |

#### 代码层根系

```
目前：DocGovernAgent → write_file → docs/auditing/（流程终止）

应当：DocGovernAgent → observer.emit({ type: governance.audit, ... })
  → PipelineObserver 分流
    → ButlerAgent 按轨分发（P0→DECISION_REQUIRED，P1+→WARNING）
    → 用户响应
    → 响应结果归档至 MemoryType.Governance
```

#### 修复依赖链

```
发现 2-B 修复 ← 依赖 ObservableEvent 新增 governance 事件类型
  ← 依赖 ButlerAgent 三轨分发策略的 governance 规则
  ← 依赖 PipelineObserver 不修改 emit-only 单向约束（§8 退化防线）
```

**建议**：在 Core-2 启动前，至少完成：
1. ObservableEvent 新增 `governance.audit` / `governance.constitution_check` 等治理事件类型
2. DocGovernAgent 在审计完成后 emit 治理事件（而非仅写磁盘）
3. ButlerAgent 三轨分发策略中新增治理事件的映射规则

这部分改动量较小（~50 行），但架构意义重大——治理层从「静默审计」升级为「用户可见」。

### 🔴 发现 3-A：审计闭环缺失 —— Core-2 阶段门禁的前置条件

**当前状态**：⏳ AM-2026-0606-002 已提案，待裁决。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **阶段门禁（3-C）** | 审计闭环是阶段门禁的输入条件。AM-2026-0606-002 定义了「未关闭的 P0 审计发现不得进入下一阶段门禁」——若审计闭环不落地，阶段门禁无法有效执行 | 🔴 高 |
| **Core-2 治理组件入宪质量** | IncidentEscalator、ContractEnforcer、CircuitBreaker 入宪前需经审计。若审计不闭环，入宪提案可能带着未解决的合规问题进入 Core-2 | 🟡 中 |
| **DocGovernAgent 扩展** | Core-2 需要 DocGovernAgent 扩展审计范围（覆盖新 Agent + 新组件）。审计闭环缺失意味着扩展后的审计引擎仍然「只写磁盘不通知」 | 🔴 高 |
| **判例体系建立** | Core-2 治理层延伸后将产生大量新判例。判例的引用、有效期、失效条件需要审计闭环中的判例管理机制（AM-2026-0606-002 §10.1 定义了判例引用有效期） | 🟡 中 |

#### 架构根系追踪

```
审计闭环（AM-2026-0606-002 宪法定义）
  ├── 整改责任人指派 → 依赖 2-B（治理事件入管线）才能通知到责任人
  ├── 整改标准与验证 → 依赖 DocGovernAgent 二次审计
  ├── 判例引用与有效期 → 依赖 Governance 分区扩展（§9.5）
  ├── 治理门禁关联 → 依赖 3-C（阶段门禁定义）
  └── 关闭条件与签署 → 依赖开拓者裁决机制

核心发现：审计闭环的落地依赖 3 个其他缺口的修复（2-B、3-C、1-A 代码层）。
```

#### 代码层缺口

`governance-loop.ts` 的 `summarizeGovernance()` 目前只统计**修宪提案**的状态，完全不包含**审计发现**的关闭状态：

```typescript
// 当前：只统计 proposal 状态
export interface GovernanceSummary {
  pendingJudgment: number;   // 待评判的提案数
  approved: number;          // 已通过待执行的提案数
  blocked: number;           // 已被阻塞的提案数
  applied: number;           // 已应用的提案数
  judgments: BatchJudgment[]; // 逐条提案的评判结果
}

// 缺少：
// openFindings: number;     // 未关闭的审计发现数
// closedFindings: number;   // 已关闭的审计发现数
// blockingFindings: number; // 阻塞门禁的审计发现数
```

#### Core-2 启动的阻塞条件

根据 AM-2026-0606-002 的宪法定义，审计闭环是阶段门禁的输入条件。这意味着：
1. Core-1 阶段的审计发现必须全部关闭（或 P1+ 经开拓者特批）
2. 审计闭环的基础设施（审计发现跟踪、状态流转、关闭裁决）必须在 Core-2 启动前就绪

**建议**：AM-2026-0606-002 的裁决通过后，代码层需要同步跟进——在 `governance-loop.ts` 中新增审计发现跟踪逻辑，与提案跟踪并列。

### 🟡 发现 3-B：宪法/治理层设计层级冲突原则缺失 —— Core-2 治理层扩展的核心风险

**当前状态**：❌ 未修复，无提案生成。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **Core-2 治理层大幅扩展** | Core-2 将引入常设委员会、临时委员会、监理实体、TrustAgent 等大量治理层组件。治理层设计文档的内容量预计增长 3-5 倍。宪法和治理层设计之间的冲突概率指数上升 | 🔴 高 |
| **超前设计与落地差异** | 治理层设计包含大量超前设计（§3.1-§3.11）。Core-2 落地过程中，超前设计与宪法现有条款必然出现不对齐——没有冲突解决原则，裁决无依据 | 🔴 高 |
| **监理封驳权的宪法依据** | 监理从取向开关升级为独立实体后，其封驳权需要宪法授权。若治理层设计定义的封驳范围与宪法 §7 确认门的安全边界重叠或冲突，无原则可依 | 🟡 中 |
| **委员会决议 vs 宪法条款** | 常设委员会的审计决议若与宪法某条款精神一致但表述不同，应以谁为准？无宪法级冲突原则 = 委员会可能做出违宪决议而不自知 | 🔴 高 |

#### 冲突场景举例（Core-2 典型）

```
场景：常设委员会审计发现 TrustModel 的自动衰减机制
     与宪法原则一「确认在用户手里」精神冲突——
     用户未确认，信任分自动衰减。

宪法立场：原则一不可变，任何用户未确认的状态变更都违宪
治理层设计：TrustModel 自动衰减是基础设施行为，不视为「操作」

冲突：两个层级的规则冲突。没有宪法级冲突原则，
     委员会不知道按哪个执行。
```

#### 修复建议

在 Core-2 启动前，宪法应新增冲突解决原则（可参考此前已有的讨论）：
1. **宪法优先原则**：宪法条款效力高于治理层设计。冲突时以宪法为准
2. **标注义务**：治理层设计中与宪法存在潜在冲突的内容须显式标注「与宪法 §X 可能存在冲突」
3. **裁决通道**：层级冲突由 DocGovernAgent 识别 → 昔涟评判 → 开拓者裁决。裁决结果记入宪法修正记录
4. **政府可演进**：冲突不意味着宪法错了——治理层设计可以先改（政府可演进），宪法后修（国家结构稳定）

### 🔴 发现 3-C：阶段门禁宪法定义不完整 —— Core-2 最直接的阻塞风险

**当前状态**：❌ 未修复。AM-2026-0606-002 部分涉及（审计发现与门禁的关联），但完整的阶段门禁定义仍缺失。

#### 对 Core-2 的影响

| 影响维度 | 影响分析 | 风险 |
|---------|---------|------|
| **Core-1→Core-2 跃迁** | 阶段门禁是 Core-1 到 Core-2 的「检查站」。没有宪法定义的门禁通过/失败标准，阶段跃迁无依据——谁来判断条件满足？谁来裁决？ | 🔴 阻塞 |
| **Core-2 准入条件不明确** | Core-2 启动需要满足哪些条件？宪法 §13 阶段模型仅描述性说明，未定义具体的检查项目和通过标准 | 🔴 阻塞 |
| **治理门禁关联（3-A）** | AM-2026-0606-002 定义的「未关闭 P0 审计发现不得进入下一阶段门禁」需要阶段门禁本身有宪法定义才能生效 | 🔴 高 |
| **Core-2→Core-3 门禁预留** | 阶段门禁不应只为 Core-1→Core-2 设计，还需为后续阶段预留扩展空间 | 🟡 中 |

#### 当前宪法 §13 的内容（摘要）

```
十三、阶段模型

Core-1：协约化与稳固化——物理归位收束
Core-2：安全与治理——Sentinel + TrustModel + 技能系统 + 新 Agent 落地
Core-3：生态与协作——多用户 / 多实例 / 联邦治理

每个阶段有阶段门禁检查表，通过后方可进入下一阶段。
```

宪法 §13 只有描述性说明，**没有**：
- 门禁检查项目的具体清单
- 每个检查项的通过/失败标准
- 检查的执行者（DocGovernAgent？常设委员会？开拓者？）
- 失败后的补救流程
- 特批机制（哪些项可以特批？谁有特批权？）
- 检查结果的归档路径

#### Core-2 启动前的必备宪法定义

```
阶段门禁（建议宪法新增 §13.1）
  ├── 检查项清单（至少包含）：
  │   ├── 宪法缺口状态：所有 P0 审计发现已关闭（或经开拓者特批）
  │   ├── 代码质量：CI gate 通过、类型检查通过、测试覆盖率 ≥ 阈值
  │   ├── 治理层就绪：审计闭环已落地、通知管线就绪
  │   ├── Core-2 组件预实现：关键组件代码已就位
  │   └── 开拓者确认：用户（开拓者）明确同意进入下一阶段
  ├── 通过/失败标准：全部检查项通过 = 通过；任一未通过 = 阻塞
  ├── 特批机制：开拓者可特批单项未通过项（但 P0 审计发现不可特批）
  ├── 执行者：DocGovernAgent 执行检查 → 昔涟评判 → 开拓者裁决
  ├── 补救流程：阻塞后 7 天内整改→重新检查；超期未整改自动升级
  └── 归档：检查结论写入 DocGovern 分区，标记 stage_gate_<阶段>
```

## 六、结论：Core-2 启动的架构评估

### 总体判断

**Core-2 在当前宪法缺口状态下（v2.5.13 + 3 项提案待裁决）不应直接启动。**  
需要先完成以下前提条件：

1. **宪法层**：阶段门禁定义（3-C）+ 层级冲突原则（3-B）入宪——这是 Core-2 的「交通规则」，没有规则不能上路
2. **提案裁决**：AM-2026-0606-002（审计闭环）和 AM-2026-0606-003（DECISION_REQUIRED 回退）完成裁决——这两项是 Core-2 治理组件运行的基础设施
3. **代码层**：审计发现跟踪（3-A 代码面）+ 治理事件入管线（2-B）落地——治理层从「静默审计」升级为「用户可见」

### 风险最大的依赖链

```
阶段门禁定义缺失 (3-C)
  → Core-1→Core-2 跃迁无依据
  → 审计闭环 (3-A) 的治理门禁关联无法生效
  → 层级冲突 (3-B) 在 Core-2 治理层扩展后集中爆发
  → 治理事件 (2-B) 不入管线 → 审计闭环第一步走不通
  → DECISION_REQUIRED 回退 (2-A) 缺失 → 用户不响应时系统阻塞
  → 紧急修宪通道 (1-B) 缺失 → 安全漏洞修复延误
```

### 最直接的阻隔壁垒

**发现 3-C（阶段门禁宪法定义不完整）** 是 Core-2 最直接的阻塞风险。  
宪法没有定义「Core-1 阶段结束的标准是什么、谁来判断、谁来裁决」——这意味着 Core-2 的启动本身就是一个没有宪法依据的行为。  

> **类比**：一个国家的新政府就职前，必须先有选举法。阶段门禁就是 Cortex 的「选举法」——定义了权力移交的条件和程序。

### 最终建议

1. **短期（Core-2 启动前）**：完成 3-C（阶段门禁）+ 3-B（层级冲突）的宪法定义，裁决 AM-2026-0606-002/-003，完成 2-B（治理事件入管线）的代码改动
2. **中期（Core-2 初期）**：完成 1-A（CHECK_ORDER 可扩展化）代码层跟进，完成 1-B（紧急修宪通道）宪法定义
3. **长期（Core-2 运行期）**：完成 2-C（通知持久化追踪）入宪，持续补充阶段门禁检查项的细化定义
