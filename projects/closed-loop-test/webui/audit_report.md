#### 2.2.2 安全标准审计

**核心发现：使用 `new Function()` 执行用户输入构成安全风险。**

```javascript
const result = new Function('return (' + sanitized + ')')();
```

| # | 违规项 | 违反条款 | 风险等级 | 说明 |
|---|--------|---------|---------|------|
| SEC-01 | **`new Function` 执行用户输入** | 宪法 §七（确认门安全原则） | **P0** | 虽然通过了字符白名单过滤（`/^[0-9+\-*/().]+$/`），但 `new Function` 本质等价于 eval。绕过正则的可能性存在（如利用 JS 引擎解析差异）。Cortex 安全原则要求所有用户输入执行必须经过 ConfirmGate 确认路径 |
| SEC-02 | **使用 `innerHTML` 设置用户可见内容** | 宪法 §七（L1 可逆写入确认） | **P2** | `resultDiv.innerHTML = '<span class="error">...'` 虽然当前内容不包含用户输入，但模式上容易引入 XSS。应使用 `textContent` + 结构化 DOM 操作 |
| SEC-03 | **错误信息泄露内部细节** | 治理层设计 §1.1（SafeErrorReporter） | **P2** | `e.message` 直接展示给用户，可能泄露引擎实现细节。应符合原则五，走 PipelineObserver 统一错误上报 |
| SEC-04 | **缺乏输入长度限制** | 安全最佳实践 | **P2** | 未限制表达式输入长度，可能被用于长字符串拒绝服务攻击 |
| SEC-05 | **未使用 Content-Security-Policy** | 安全最佳实践 | **P3** | HTML 头部缺少 CSP 头，无法防范内联脚本注入 |

**关于 `new Function` 的补充分析**：

当前过滤正则 `^[0-9+\-*/().]+$` 允许以下字符：
- 数字 `0-9`
- 运算符 `+`, `-`, `*`, `/`
- 括号 `(`, `)`
- 小数点 `.`

理论上这些字符无法构造函数调用或对象访问。但考虑到：
1. 正则白名单是唯一防线——任何正则绕过漏洞都会导致完全代码执行
2. JavaScript 引擎对 Unicode 字符的归一化可能导致绕过（如全角字符 `.`）
3. Cortex 确认门机制要求 L1 以上操作必须经用户确认——此处直接执行无确认

**建议替代方案**：使用 AST 解析器（如 `mathjs` 或手写递归下降解析器）替代文本执行。

## 三、判例引用

### NG-2026-0509-Persist-False-Positive（NG 判例）

> 持久化操作不允许假阳性。若操作失败必须传播为失败。

**适用说明**：`test.html` 中 `new Function()` 调用若抛出异常，虽被 `try-catch` 捕获并显示了错误消息，但错误信息直接暴露给用户端，未遵循 SafeErrorReporter 三档上报协议。此判例要求所有错误必须通过统一管道上报。

### D-01~D-05（自审视权限偏差判例群）

> 自审视模式下 Agent 工具权限临时提升至 FULL_TOOLSET，但写入硬约束于 `test-output/self-examination-soft/`。

**适用说明**：`webui/` 目录下的文件不在自审视的 `test-output/` 范围内，但它们代表了另一种形式的"游离资产"——未纳入任何包结构、未通过任何门禁、未受任何审计约束。此判例群确立的原则是：**不受约束的代码是危楼**。

## 四、整改建议

### 4.1 处置优先级

| 优先级 | 项 | 时限 |
|--------|----|------|
| **P0** | `new Function()` 替换为 AST 解析器 | 当前门禁前 |
| **P1** | 删除 `test.txt`、删除或实现 `calculator.js` | 当前门禁前 |
| **P2** | `innerHTML` 替换、CSP 头部、输入长度限制 | 下一迭代 |
| **P3** | TypeScript 迁移、ESLint 纳入、`@ci` 标签标注 | 长期 |

### 4.2 具体修复方向

1. **`test.html` 安全修复**：
   - 将 `new Function('return (' + sanitized + ')')()` 替换为数学表达式 AST 解析器
   - 移除 `innerHTML` 改用 `textContent` + `createElement`
   - 添加 `Content-Security-Policy` 元标记
   - 添加输入长度限制（建议 ≤ 256 字符）
   - 错误消息改为用户友好的通用提示，细节走 console.error（需符合 PipelineObserver 规范）

2. **`calculator.js` 处置**：
   - 选项 A：实现完整计算器（可测试、可审计、可集成）
   - 选项 B：删除，将功能合并至 `test.html`
   - 无论哪种，需标注 `@ci: manual` 标签

3. **`test.txt` 处置**：
   - 立即删除。如需要测试文本文件读写，请使用 `packages/engine/tests/fixtures/` 下的结构化测试数据

### 4.3 治理建议

**建议一**：`webui/` 目录应设立治理门禁。当前该目录是"治理盲区"——没有 Agent 审计、没有阶段门禁检查、没有宪法一致性验证。建议将 `webui/` 纳入 DocGovernAgent 的 `doc_audit` 扫描范围。

**建议二**：新增 `webui/` 的准入规范：
- 所有文件必须有文件头注释（用途、作者、版本）
- 所有 JS 文件必须使用 TypeScript（或至少 JSDoc 完整标注类型）
- 必须通过 ESLint 检查
- 必须标注 `@ci: manual` 标签（因 webui 非核心引擎代码）
- HTML 文件必须包含 CSP 头部

**建议三**：将本次审计报告归档至 DocGovern 分区（`doc-govern/`），供后续阶段门禁引用。
